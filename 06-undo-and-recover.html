6  撤销、改错与找回

很多人学 Git 时最怕的，不是提交，而是改错以后不知道怎么退。文件删了怎么办？刚才那次提交写错了怎么办？不小心把东西加进暂存区又怎么办？

这一章专门处理这些“做错了以后怎么办”的场景。重点不是教你一招万能回退，而是教你先分辨：你现在想撤销的是工作区、暂存区，还是已经提交进历史的内容。

本章会讲：

  • git restore
  • git reset
  • git revert
  • 修改最后一次提交
  • 回退时最容易踩到的坑

这一章读完以后，你对 Git 的恐惧会下降很多，因为你会知道，大多数错误都不是世界末日。

6.1 先记住一个总原则

Git 里很多“回退”操作看起来很像，真正的区别在于：你到底想动哪一层。

先把问题拆成三类：

  1. 工作区改错了，但还没准备提交
  2. 已经 add 进暂存区了，但还没提交
  3. 已经提交进历史了，现在后悔了

这三类情况，用的命令不一样。如果还没先分清楚自己现在在哪一类，就很容易乱按命令。

6.2 先准备一个练习仓库

为了安全起见，这一章最好在测试仓库里练，不要一开始就在正式项目上尝试。

先进入一个已有提交历史的仓库，然后执行：

git status
git log --oneline -3

6.2.1 这一步怎样算成功

算成功：

  • git status 能正常执行
  • git log --oneline -3 能看到最近几笔提交

算失败：

  • 你还不在 Git 仓库里
  • 仓库里还没有任何提交，后面的历史相关操作就没法练

6.3 第一类：工作区改错了，还没 add

这是最轻的一种情况。假设你打开 README.md，加了一段内容，后来发现完全写错了。

先执行：

git status
git diff

如果你确认这段修改不要了，可以执行：

git restore README.md
git status

6.3.1 git restore 在做什么

这条命令的意思是：把工作区里的这个文件，恢复成暂存区或上一次提交里记录的样子。对初学者来说，先把它理解成“撤销还没提交的工作区修改”就够了。

6.3.2 你现在应该看到什么

如果这次修改还没 add，执行完 git restore README.md 以后：

  • git status 里不该再显示这个文件是 modified
  • 你刚才那段没保存到历史的修改会消失

6.3.3 这一步怎样算成功

算成功：

  • 你确认了工作区改动真的被撤销
  • 文件回到之前那一版
  • git status 不再显示这次修改

算失败：

  • 你其实还没看 diff，就把改动直接撤掉了
  • 你以为是在撤销一小部分，结果整份文件未提交修改都被恢复了

这也是为什么我一直建议先看 git diff 再做撤销。先确认不要，再动手。

6.4 第二类：已经 add 了，但还没提交

这是更常见的场景。你刚才手快执行了：

git add README.md
git status

然后才发现：这次修改还不想交，或者这一版还没整理好。

这时问题分两层：

  • 我想先把它从暂存区拿出来
  • 我想不想连工作区内容一起丢掉

这两件事不是一回事。

6.4.1 只想撤销暂存，不想丢掉文件内容

如果你只是想把它从“准备提交”改回“还在修改中”，可以执行：

git restore --staged README.md
git status

6.4.2 你现在应该看到什么

执行完以后，常见情况会变成：

  • README.md 不再位于 Changes to be committed
  • 它重新出现在“还没暂存的修改”里

也就是说，内容还在，只有“准备提交”这个状态被撤回了。

6.4.3 这一步怎样算成功

算成功：

  • 你成功把文件从暂存区拿出来
  • 文件内容没有丢
  • git diff 还能看到你的修改

算失败：

  • 你本来只想撤销暂存，却把工作区内容也一起清掉了
  • 你分不清“内容还在”和“已经准备提交”这两件事

对初学者来说，这一步非常重要，因为它能帮助你真正把工作区和暂存区分开。

6.4.4 想连工作区内容一起不要

如果你既不想让它留在暂存区，也不想保留工作区里的内容，可以这样做：

git restore --staged README.md
git restore README.md
git status

先把它从暂存区拿出来，再把工作区恢复掉。分两步做，最不容易混乱。

6.5 第三类：已经提交了，但提交说明写错了

这是很常见、也相对安全的一种情况。比如你刚提交了：

git commit -m "update"

结果马上觉得这个说明太空，想改成更清楚一点。

这时可以执行：

git commit --amend -m "补充练习说明"
git log --oneline -1

6.5.1 --amend 在做什么

它的作用是：修改最后一笔提交。最常见用途有两个：

  • 改最后一次提交说明
  • 把刚才漏掉的少量内容补进最后一次提交

6.5.2 你现在应该看到什么

执行成功后：

  • git log --oneline -1 会显示新的提交说明
  • 最后一笔提交的编号通常也会变化

6.5.3 这一步怎样算成功

算成功：

  • 你成功改掉了最后一次提交说明
  • 日志里最新一笔已经变成新的标题

算失败：

  • 你以为只是改标题，结果其实把别的内容也混进去
  • 你在已经推送给别人协作的提交上随意改写历史

这里要先提醒一件事：git commit --amend 会改写最后一笔提交历史。自己本地刚做完、还没共享出去时通常问题不大；如果这笔提交已经推送给别人，后面要更小心。

6.6 第四类：已经提交了内容，但现在想“撤回一笔提交”

这里要非常分清楚：你到底是想“改写历史”，还是“保留历史，但再做一笔反向修改”。

对初学者来说，先学稳 git revert，比急着碰复杂的重写历史更安全。

6.6.1 用 git revert 撤回一笔提交

先看最近日志：

git log --oneline -3

假设最近一笔提交编号是：

abc1234 补充练习说明

现在执行：

git revert abc1234
git log --oneline -4

6.6.2 git revert 在做什么

它不是把那笔提交从历史里抹掉，而是新增一笔“反向提交”，把那次改动抵消掉。

这点很重要，因为它比粗暴改写历史更安全，也更适合已经共享出去的历史。

6.6.3 你现在应该看到什么

执行成功后：

  • 日志里不会少掉原来的那笔提交
  • 反而会多出一笔新的“revert”提交
  • 文件内容会回到那笔提交之前的状态

6.6.4 这一步怎样算成功

算成功：

  • 你成功撤销了某次提交带来的效果
  • 历史仍然完整，没有直接消失
  • git log 里能看到原提交和新的撤销提交

算失败：

  • 你以为 revert 会让历史直接消失
  • 你看到多了一笔新提交，就误以为操作失败

很多初学者第一次看到“怎么又多一笔提交”会疑惑，但这正是 revert 的正常行为。

6.7 git reset 应该怎么理解

git reset 的功能更宽，也更容易让人混乱。初学阶段，你可以先把它拆成两种最常见用途：

  • 把暂存区退回去
  • 把分支指针退回到更早的提交

前一种你前面已经通过 git restore --staged 练过了。后一种如果直接配合 --hard 用，风险会高很多。

所以对初学者来说，先记住这个原则：

  • 日常撤销暂存，优先用 git restore --staged
  • 想安全撤销一笔已提交改动，优先用 git revert
  • git reset --hard 不要在还没完全看懂前乱用

6.8 一个高风险命令为什么要小心

像下面这种命令：

git reset --hard HEAD~1

它的效果通常是：

  • 把当前分支直接退回上一笔提交
  • 同时把工作区和暂存区也一起重置过去

这很强，也很危险。因为如果你还没想清楚自己在做什么，工作区里没保存的内容可能会直接消失。

6.8.1 这类操作怎样判断成功和失败

算成功：

  • 你非常明确自己是在测试仓库或可承受场景里操作
  • 你知道它会同时影响历史、暂存区和工作区

算失败：

  • 你把它当成“万能撤销键”
  • 你还没看清当前状态就直接运行
  • 运行完才发现本地未保存内容没了

这本书后面会再讲更细的历史整理，但在入门阶段，我更希望你先形成安全直觉，而不是先学最猛的命令。

6.9 一个更稳的判断顺序

每次想“回退”时，先不要急着敲命令，先问自己下面三个问题：

  1. 我想撤销的是工作区、暂存区，还是历史
  2. 我是想把内容留着，只改状态，还是连内容一起不要
  3. 这笔历史是不是已经共享给别人了

很多 Git 错误，本质上不是命令不会，而是没先回答这三个问题。

6.10 本章练习

6.10.1 练习 1：撤销工作区改动

  1. 修改一个文件
  2. git status
  3. git diff
  4. git restore <文件名>
  5. git status

算成功：

  • 你能撤销未暂存的改动
  • 你知道文件为什么恢复了

算失败：

  • 你没先确认内容，就直接把修改丢掉

6.10.2 练习 2：撤销暂存但保留内容

  1. 修改一个文件
  2. git add <文件名>
  3. git status
  4. git restore --staged <文件名>
  5. git status
  6. git diff

算成功：

  • 文件从暂存区退回工作区
  • 内容还在

算失败：

  • 你想保留内容，却把内容也一起清掉了

6.10.3 练习 3：修改最后一次提交说明

  1. 做一笔提交，故意写个不好的说明
  2. git commit --amend -m "新的说明"
  3. git log --oneline -1

算成功：

  • 你能看见最后一次提交说明被改掉

算失败：

  • 你不知道 amend 改的是哪一笔提交

6.10.4 练习 4：用 revert 撤销一笔提交

  1. 新做一笔简单提交
  2. 记录它的提交编号
  3. 执行 git revert <提交编号>
  4. 查看 git log --oneline -4

算成功：

  • 你能看到原提交还在
  • 新增了一笔撤销提交
  • 内容效果被抵消了

算失败：

  • 你以为撤销就是把历史抹掉
  • 看到多出一笔提交就以为出错
