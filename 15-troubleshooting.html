15  常见问题与排查

学 Git 的过程里，真正拖慢人的往往不是概念，而是那些突然冒出来的错误信息。明明只是想提交一下，结果出现“有冲突”“分支领先”“远端拒绝推送”之类的提示，很容易一下就乱掉。

这一章不会追求面面俱到，而是整理初学者最常遇到的几类问题，并教你先判断问题在哪一层，再决定下一步怎么做。

本章会整理这些场景：

  • 文件改了却没被追踪
  • 提交了但好像不在预期分支上
  • 推送被拒绝
  • 拉取以后出现冲突
  • 想回到之前某个版本却不知道该用哪条命令

这一章的重点不是让你死记硬背报错，而是慢慢建立一种排查顺序。

15.1 先建立一个排查顺序

很多人一看到 Git 报错，就急着搜索一条命令复制过去。这样有时能碰巧解决，但更容易把问题越弄越乱。

更稳的顺序是先问四个问题：

  1. 我现在在哪个仓库、哪个分支
  2. 工作区干不干净
  3. 修改停在工作区、暂存区，还是已经进历史了
  4. 问题发生在本地，还是发生在和远端同步时

这四个问题比任何单条“万能命令”都更重要。

15.2 排查前最值得先跑的几条命令

遇到问题时，先不要乱动，先跑这几条：

git status
git branch
git log --oneline -5
git remote -v

15.2.1 这一步怎样算成功

算成功：

  • 你先把当前状态看清楚了
  • 你知道自己在哪个分支上
  • 你知道最近历史长什么样
  • 你知道当前仓库有没有远端

算失败：

  • 你还没看状态，就先乱试命令
  • 你连自己现在在哪个分支上都不清楚

15.3 问题 1：文件明明改了，为什么 Git 看不到

15.3.1 常见现象

你明明修改了一个文件，但：

  • git status 里看不到它
  • 或者你以为改了，git diff 却什么都没显示

15.3.2 先检查什么

先检查下面几件事：

pwd
ls
git status

15.3.3 常见原因

  • 你改的不是当前仓库里的那一份文件
  • 文件根本还没保存到磁盘
  • 这个文件被 .gitignore 忽略了
  • 你已经 git add 过，所以 git diff 看不到工作区变化

15.3.4 进一步怎么判断

如果怀疑是被忽略了，可以看看目录里有没有 .gitignore，再检查规则。

如果怀疑是已经暂存了，可以执行：

git diff
git diff --staged

15.3.5 这类问题怎样算成功解决

算成功：

  • 你已经找到文件为什么没出现在预期位置
  • 你知道问题出在路径、保存、忽略规则，还是暂存状态

算失败：

  • 你还没判断原因，就开始怀疑 Git 坏了

15.4 问题 2：为什么我刚提交的东西不在我以为的分支上

15.4.1 常见现象

你刚做了一笔提交，结果后来发现：

  • 这笔提交不在你以为的任务分支上
  • 或者它其实进了主分支

15.4.2 先检查什么

先执行：

git branch
git log --oneline -5

15.4.3 常见原因

  • 你开始修改前没有先确认当前分支
  • 你以为已经 git switch 过去了，其实没有

15.4.4 这类问题怎样算成功解决

算成功：

  • 你已经确认提交现在到底在哪个分支
  • 你没有继续在错误分支上追加更多修改

算失败：

  • 你发现站错分支了，却还继续往下做

这种情况下一般不要慌，先停下来，看清楚当前位置，再决定是继续整理、还是把改动移到别处分支上。

15.5 问题 3：git diff 为什么什么都不显示

15.5.1 常见现象

你明明知道自己改了内容，但：

git diff

却没有任何输出。

15.5.2 常见原因

最常见的原因有两个：

  • 你其实还没改成功，文件内容和上次一样
  • 你已经 git add 过了，工作区和暂存区现在一样

15.5.3 先怎么判断

执行：

git status
git diff
git diff --staged

15.5.4 这类问题怎样算成功解决

算成功：

  • 你能判断修改现在到底是在工作区还是暂存区
  • 你知道为什么某一条 diff 没输出

算失败：

  • 你只看见没输出，就以为改动消失了

15.6 问题 4：提交时提示没有可提交内容

15.6.1 常见现象

你执行：

git commit -m "..."

结果 Git 提示没有内容可提交。

15.6.2 常见原因

  • 你改了文件，但还没 git add
  • 你以为自己改了，实际上文件没保存
  • 你已经提交过了，现在工作区其实是干净的

15.6.3 先怎么判断

执行：

git status
git diff
git diff --staged

15.6.4 这类问题怎样算成功解决

算成功：

  • 你已经看清楚内容到底停在哪一层
  • 如果需要，你能补上正确的 git add

算失败：

  • 你没看状态，就一遍遍重复敲 git commit

15.7 问题 5：推送被拒绝

15.7.1 常见现象

你执行：

git push

结果被远端拒绝。

15.7.2 先检查什么

先看：

git branch -vv
git remote -v
git status

15.7.3 常见原因

  • 当前分支还没和远端建立跟踪关系
  • 远端已经有更新，你本地还没同步
  • 认证失败
  • 你推错了远端或分支

15.7.4 这类问题怎样算成功解决

算成功：

  • 你知道到底是跟踪关系问题、同步问题，还是认证问题
  • 你不是一看到拒绝就乱换命令

算失败：

  • 你连拒绝发生在哪个层面都没分清楚

15.8 问题 6：git pull 之后为什么内容变了，甚至出现冲突

15.8.1 常见现象

你只是想同步一下远端，结果：

  • 本地文件变了
  • 甚至出现冲突

15.8.2 原因是什么

因为 pull 通常不是纯下载，它是“取回远端变化，并尝试合并到当前分支”。

所以一旦远端也改了你本地正在改的地方，就可能带来冲突。

15.8.3 先怎么判断

执行：

git status
git branch
git log --oneline -5

如果出现冲突，再打开具体文件看有没有这些标记：

  • <<<<<<<
  • =======
  • >>>>>>>

15.8.4 这类问题怎样算成功解决

算成功：

  • 你知道 pull 之后发生的是“同步并整合”
  • 遇到冲突时，你能回到冲突章节那套流程去处理

算失败：

  • 你把 pull 理解成只读动作
  • 一出现冲突就以为仓库坏了

15.9 问题 7：想撤销，但不知道该用 restore、revert 还是别的

15.9.1 先不要先问命令，先问位置

你真正该先问的是：

  • 这个改动还在工作区吗
  • 已经进暂存区了吗
  • 已经提交进历史了吗

15.9.2 一个最小判断表

如果是下面这些情况：

  • 还没 add，只在工作区 常见先看 git restore
  • 已经 add，但还没 commit 常见先看 git restore --staged
  • 已经 commit 了，想安全抵消效果 常见先看 git revert
  • 只是最后一次提交说明写错 常见先看 git commit --amend

15.9.3 这类问题怎样算成功解决

算成功：

  • 你先分清当前改动停在哪一层
  • 然后再选对应工具

算失败：

  • 你一看到“想撤销”，就直接去碰最猛的命令

15.10 问题 8：远端、分支、Fork 关系全乱了

15.10.1 常见现象

你已经开始用 GitHub 协作，但现在说不清：

  • 当前分支是什么
  • origin 指向哪里
  • upstream 指向哪里
  • 当前 PR 到底是往哪边发

15.10.2 先检查什么

执行：

git branch
git branch -vv
git remote -v

15.10.3 这类问题怎样算成功解决

算成功：

  • 你重新看清了当前分支和远端关系
  • 你知道自己现在是在和哪个仓库打交道

算失败：

  • 你关系还没理清，就继续推、继续拉、继续发 PR

15.11 一个更稳的排查顺序

以后再遇到问题时，可以优先按这个顺序来：

  1. 先看 git status
  2. 再看 git branch
  3. 再看最近日志 git log --oneline -5
  4. 如果涉及远端，再看 git remote -v
  5. 如果涉及改动内容，再看 git diff 和 git diff --staged

这套顺序不是最快，但对初学者来说通常最稳。

15.12 本章练习

15.12.1 练习 1：自己诊断一次“看不到修改”

故意制造一个小问题，例如：

  • 改一个文件后先 git add
  • 然后再去看 git diff

接着用下面这些命令判断发生了什么：

git status
git diff
git diff --staged

算成功：

  • 你能解释为什么某条命令没输出

算失败：

  • 你只会说“Git 没反应”

15.12.2 练习 2：自己诊断一次“推送失败”

即使你这次没有真的失败，也请你先回答：

  1. 推送失败时我会先看哪几条命令
  2. 我怎么判断是分支问题、远端问题，还是认证问题

算成功：

  • 你已经有一套先看状态、再分类的问题顺序

算失败：

  • 你一想到推送失败，脑子里只有“试试看别的命令”

15.12.3 练习 3：给自己做一张最小排查卡片

把下面几条抄成你自己的排查顺序：

git status
git branch
git log --oneline -5
git remote -v
git diff
git diff --staged

算成功：

  • 你已经不再把 Git 排查理解成“背一堆报错”
  • 而是知道先看哪里，再决定下一步

算失败：

  • 你仍然希望有一条命令能一次性解决所有 Git 问题
