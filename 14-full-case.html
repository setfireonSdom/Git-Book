14  一个完整案例

前面的章节把工具和概念拆开讲，这一章会把它们重新串起来。我们会用一个完整的小项目走一遍从建立仓库到协作合并的过程，让前面学过的动作不再零散。

这个案例改用“个人网站”来演示，因为它常见、够轻，也方便把注意力放回 Git 本身。同时，它也能自然覆盖单人修改、分支、GitHub、Pull Request、合并和标签。

本章会特别强调两件事：

  • 每个 Git 动作是在解决什么具体问题
  • 当流程真的串起来时，哪些地方最容易乱

读完这一章，你会更清楚这些命令不是一颗颗散的珠子，而是一条完整工作流。

14.1 这个案例要做到什么

这一章不是再教一批新命令，而是把前面已经学过的东西串成一条完整路线。

做到最后时，你应该能看清楚这条线：

  • 从零开始建立一个个人网站仓库
  • 持续提交本地修改
  • 用分支处理新任务
  • 把仓库接到 GitHub
  • 用 Pull Request 做一次协作
  • 合并后整理主线
  • 给一个稳定节点打版本标签

如果这一章你能顺着走完，前面那些分散章节就会真正接起来。

14.2 场景设定

我们现在假设你正在做一个个人网站，目录里有这些文件：

  • index.html
  • about.html
  • style.css

你一开始是一个人做，后来请朋友帮你补一个“项目经历”页面。这正好能把 Git 和 GitHub 的主线都串起来。

14.3 第一阶段：自己先把项目立起来

先在本地准备一个目录：

mkdir personal-site-demo
cd personal-site-demo
git init
printf "<h1>我的个人主页</h1>\n" > index.html
printf "<h1>关于我</h1>\n" > about.html
printf "body { font-family: sans-serif; }\n" > style.css
git status

14.3.1 这一步在解决什么问题

这一段不是为了“先敲几条命令”，而是为了把一个普通网站项目正式交给 Git 管。你要从“普通文件夹”进入“有历史的仓库”。

14.3.2 你现在应该看到什么

你通常会看到：

  • 当前目录已经是 Git 仓库
  • index.html、about.html、style.css 这些文件还没被追踪

14.3.3 这一步怎样算成功

算成功：

  • 个人网站目录已经初始化为仓库
  • Git 能看到这几个新文件

算失败：

  • 文件建出来了，但你其实没在仓库目录里
  • git status 里根本看不到刚才建的文件

14.4 第二阶段：做第一笔提交，让项目有起点

现在把这批初始文件交进去：

git add index.html about.html style.css
git commit -m "建立个人网站骨架"
git log --oneline -3

14.4.1 这一步在解决什么问题

到这里，你不是单纯把文件存一下，而是给这个项目留下了一个明确起点。以后不管改成什么样，你都知道“最早的骨架”是哪一笔提交。

14.4.2 这一步怎样算成功

算成功：

  • 仓库里已经有第一笔正式历史
  • git log --oneline 能看到 建立个人网站骨架

算失败：

  • 你以为提交成功了，其实日志里并没有这笔记录
  • 你没有建立历史，就直接往后堆修改

14.5 第三阶段：日常修改，保持小步提交

现在你继续完善网站。比如补充首页和关于页简介：

printf "\n<p>这里展示我的作品、经历和联系方式。</p>\n" >> index.html
printf "\n<p>我是一名前端初学者，正在练习 Git 和 GitHub。</p>\n" >> about.html
git status
git diff

然后把这次修改整理成一笔提交：

git add index.html about.html
git commit -m "补充首页和关于页简介"
git log --oneline -3

14.5.1 这一步在解决什么问题

这里体现的是最基本也最重要的节奏：改完以后先看，再决定这次提交交什么。不是写很久才一次性交，而是小步、清楚地往前走。

14.5.2 这一步怎样算成功

算成功：

  • 本地历史里又多了一笔主题明确的提交
  • 你能说清楚这一笔提交在做什么

算失败：

  • 你一口气把一堆互不相干的修改全塞进一笔提交
  • 提交说明太空，过几天自己也看不懂

14.6 第四阶段：遇到新任务时，不要直接改主线

现在朋友建议你加一个“项目经历”页面。这时不要直接在主分支上写，先开分支：

git switch -c add-projects-page
printf "<h1>项目经历</h1>\n" > projects.html
git add projects.html
git commit -m "新增项目经历页面骨架"
git branch

14.6.1 这一步在解决什么问题

你现在不是在做“已经确认收进主线”的内容，而是在处理一个新任务。分支的作用，就是把这个任务先隔离出来。

14.6.2 这一步怎样算成功

算成功：

  • 新任务发生在独立分支上
  • 主分支暂时还保持原状

算失败：

  • 你忘了切分支，结果又把实验性内容直接做进主线
  • 你以为自己在分支上，其实一直没切出去

14.7 第五阶段：把仓库接到 GitHub

现在本地项目已经写到一个像样的阶段了，可以接 GitHub。

先在 GitHub 上建一个空仓库，然后在本地执行：

git remote add origin https://github.com/yourname/personal-site-demo.git
git remote -v
git branch

确认远端和当前分支以后，把主分支推上去。假设主分支名是 main：

git switch main
git push -u origin main

然后再把任务分支也推上去：

git switch add-projects-page
git push -u origin add-projects-page

14.7.1 这一步在解决什么问题

到这里，你就不只是有一个本地网站仓库了。现在这个项目已经有远端副本，可以开始进入协作。

14.7.2 这一步怎样算成功

算成功：

  • GitHub 页面已经能看到主分支内容
  • 任务分支也已经出现在远端

算失败：

  • 你只推了主分支，却以为任务分支也已经上去
  • 你没确认当前分支名，就把错误分支推到了远端

14.8 第六阶段：发 Pull Request，请别人看

现在回到 GitHub，为 add-projects-page 发一个 Pull Request，目标是主分支。

一个最小可用的 PR 可以这样写：

  • 标题：新增项目经历页面骨架
  • 描述：添加项目经历页面，为后续补充作品说明和链接做准备。

14.8.1 这一步在解决什么问题

PR 的作用不是“告诉别人我改完了”，而是把这次改动正式交给 review。别人可以在同一个地方看差异、提意见、讨论是否适合并进主线。

14.8.2 这一步怎样算成功

算成功：

  • PR 指向了正确的主分支
  • 标题和描述能说清这次改动在做什么

算失败：

  • PR 方向错了
  • 你发了 PR，但别人根本看不懂这次改动的目标

14.9 第七阶段：根据 review 再补一笔修改

假设朋友 review 后说：只有页面标题不够，最好再补一段简介。

你不需要重开分支，也不需要重开一个新 PR。直接在原分支继续改：

printf "\n<p>这里会继续补充我做过的练习项目。</p>\n" >> projects.html
git add projects.html
git commit -m "补充项目经历页面说明"
git push

14.9.1 这一步在解决什么问题

这一步说明一件很重要的事：PR 不是“一次性发出去就不能动”。review 之后，通常就是在同一个分支上继续补。

14.9.2 这一步怎样算成功

算成功：

  • 原 PR 页面自动更新
  • 这次补改和前面的任务仍然保持在同一条线上

算失败：

  • 你跑去别的分支修改，导致当前 PR 没有更新
  • 你以为 review 后必须重开新 PR

14.10 第八阶段：合并后回到主线

PR 通过以后，在 GitHub 上完成 merge。然后回到本地整理：

git switch main
git pull
git branch -d add-projects-page
git branch
git log --oneline -5

14.10.1 这一步在解决什么问题

合并不只是网页上点一下按钮。真正收尾时，你还要让本地主线跟上远端，并把已经完成的任务分支清掉。

14.10.2 这一步怎样算成功

算成功：

  • 主分支已经包含这次页面新增
  • 本地主分支和远端保持一致
  • 已完成分支已经清理掉

算失败：

  • 远端已经合并，本地却还停在旧状态
  • 你积了一堆旧分支，自己也不清楚哪些已经做完

14.11 第九阶段：给一个稳定节点打标签

假设这时你觉得“个人网站骨架 + 首页与关于页简介 + 项目经历页面”已经到了一个能回看的阶段，就可以打一个版本标签：

git tag v0.1
git tag
git show v0.1

14.11.1 这一步在解决什么问题

标签是在告诉未来的你：这个状态值得记下来。以后你回头找版本，不用靠模糊印象，只要看标签就行。

14.11.2 这一步怎样算成功

算成功：

  • 仓库里出现 v0.1
  • 你能确认它指向当前这个稳定节点

算失败：

  • 你打了标签，却不知道它落在哪一笔提交上
  • 你把“最新提交”和“版本节点”混成一回事

14.12 这条完整工作流里，最容易乱的地方

把整条线串起来以后，最常见的混乱通常出在下面这些地方：

14.12.1 一开始不建立清楚的本地历史

后面一旦要协作，就会发现主线本身就很乱。

14.12.2 任务来了却不切分支

这样很快就会把主线和半成品混在一起。

14.12.3 发了 PR 却没有继续在原分支上补修改

结果 review 流程断掉。

14.12.4 远端合并后，本地不整理

网页上看起来已经结束了，本地却还停在旧状态。

14.12.5 版本节点不打标签

以后回头找某个阶段，只能靠猜。

14.13 这一章真正要带走什么

读完这一章，最重要的不是再多记几条命令，而是看清这件事：

Git 和 GitHub 不是很多孤立动作的堆叠，而是一条从本地整理，到远端协作，再回到主线收尾的连续流程。

你只要把这条流程走顺，很多原本零散的命令都会自然找到位置。

14.14 本章练习

14.14.1 练习 1：自己走完一次最小案例

请用一个测试目录，从头做一遍：

  1. 建仓库
  2. 做两笔本地提交
  3. 开一个任务分支
  4. 在分支上做一笔提交
  5. 连接 GitHub
  6. 推送主分支和任务分支
  7. 发 PR
  8. 合并后回主线整理
  9. 打一个版本标签

算成功：

  • 你能把前面学过的动作串成一条连续路线
  • 你知道每一步分别在解决什么问题

算失败：

  • 单看每一章好像都懂，一串起来就不知道下一步该做什么

14.14.2 练习 2：检查自己最容易乱在哪一段

回头问自己：

  1. 我最容易乱的是本地提交节奏
  2. 还是分支和主线的关系
  3. 还是远端、PR 和 merge 之后的整理

算成功：

  • 你能指出自己现在最薄弱的那一段

算失败：

  • 你只觉得“Git 很乱”，却说不清到底乱在哪一步
