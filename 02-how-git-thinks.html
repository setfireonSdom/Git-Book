2  Git 是怎么想的

很多初学者不是记不住命令，而是脑子里没有画面。只要看不见 Git 在记录什么，add、commit、status 这些命令就会一直像零散按钮。

这一章要建立的，是 Git 最重要的心智模型。你可以先把 Git 想成一套记录修改历史的方法，但它不是看到文件就自动保存，而是让你分步骤决定：哪些修改要纳入这一次提交，哪些还不要。

本章重点包括：

  • 工作区是什么
  • 暂存区是什么
  • 仓库里的历史是什么
  • 一次提交到底保存了什么
  • 为什么 git add 和 git commit 要分开

这一章讲清楚后，后面的日常操作会轻松很多，因为你不再只是背命令，而是知道命令分别作用在哪一层。

2.1 先把这张图装进脑子里

先把 Git 想成三层：

  1. 工作区
  2. 暂存区
  3. 仓库历史

它们不是三个完全独立的世界，但你每次操作，几乎都在这三层之间移动内容。

2.1.1 工作区

工作区就是你现在手上正在改的文件。你在编辑器里打字、删句子、加代码，改的都是工作区里的内容。

它的特点是：

  • 改了就改了
  • 还没正式进入历史
  • 改到一半也很正常

工作区最像你桌上摊开的草稿。

2.1.2 暂存区

暂存区是很多人第一次学 Git 时最疑惑的地方。它像一个“准备提交清单”。你不是一改文件就直接进入历史，而是先挑出这次准备交出去的修改，放进暂存区。

它的作用是：让你把一次提交组织得更干净。

比如你今天同时做了两件事：

  • 修正了一段错别字
  • 新增了一个例子

如果你想把这两件事分成两次提交，就需要暂存区。因为 Git 不是强迫你把手头所有变化一次全交出去。

2.1.3 仓库历史

仓库历史就是已经提交完成的记录。每一次提交，都会成为这条历史上的一个节点。以后你看日志、比较差异、回到旧版本，都是在和这条历史打交道。

所以可以先这样记：

  • 工作区：你正在改
  • 暂存区：你准备交
  • 仓库历史：你已经交出去并记录下来了

2.2 为什么 git add 和 git commit 要分开

很多初学者一开始会觉得这设计有点绕，为什么不能改完直接提交。

因为现实里的修改，往往不是整块整块长得很整齐。你今天可能一边修错字，一边试新写法，一边顺手改了标题。如果改完就自动提交，你会很难把这次修改说清楚。

git add 的作用，就是先挑选“这一次要提交的内容”。

git commit 的作用，则是把已经挑好的内容正式记进历史。

这两步分开以后，你就能做到：

  • 把一团修改拆成几笔
  • 暂时不提交还没整理好的内容
  • 让提交历史以后更容易读

如果把这件事再说白一点，就是 Git 不希望你把“我今天手上所有乱七八糟的改动”原封不动扔进历史。它逼你多做一步选择，而这一步选择，恰恰决定了历史后来会不会清楚。

2.3 一次提交到底保存了什么

很多初学者会把提交想成“把整个文件重新存一份”。这个理解不算全错，但还不够准确。

更贴近的说法是：

  • 提交会记录当时项目的一个状态
  • 同时把这次状态和上一笔历史连起来

所以提交真正有价值的地方，不只是“这里存过一次”，而是“你能知道这个状态是从上一个状态怎么变来的”。

这也是为什么提交一旦写得清楚，项目历史读起来就像一条有逻辑的变化线；写得乱时，历史就会变成一团很难回看的存档。

2.4 用一个具体场景来理解三层

假设你在维护一个个人网站，当前目录里有个 index.html。

一开始，文件里写的是：

<h1>欢迎来到我的网站</h1>

你后来把它改成：

<h1>欢迎来到我的网站</h1>
<p>这里记录我的作品和联系方式。</p>

这时事情发生在工作区，还没有进入历史。

如果你接着执行：

git add index.html

这份修改就会进入暂存区。意思不是“已经永久保存”，而是“这次准备把它交出去”。

再执行：

git commit -m "补充首页简介"

它才会进入仓库历史，成为一笔正式记录。

2.5 现在就来做一次最小实操

这一章开始，我们正式动手。请在终端进入一个你打算练习的目录。为了不影响现有项目，你可以新建一个测试目录。

mkdir git-thinking-demo
cd git-thinking-demo
git init
printf "第一版内容\n" > notes.txt
git status

2.5.1 你现在应该看到什么

如果顺利，git init 之后会出现类似“Initialized empty Git repository”的提示。不同系统提示文字可能略有不同，但重点都一样：这里已经变成 Git 仓库了。

git status 这时通常会告诉你：

  • 当前在某个分支上，常见是 main 或 master
  • 有一个 notes.txt 还没有被追踪

也就是你大概率会看到类似 Untracked files 的提示。

2.5.2 这一步怎样算成功

算成功：

  • git init 没有报错
  • 当前目录里已经有 Git 仓库
  • git status 能看到 notes.txt
  • 状态里显示它还没被追踪

算失败：

  • 终端提示 git: command not found
  • git status 报错，说当前目录不是 Git 仓库
  • 你明明建了 notes.txt，状态里却没有看到它

如果失败，先检查三件事：

  1. 你电脑上有没有安装 Git
  2. 你现在所在的目录是不是刚才初始化仓库的目录
  3. notes.txt 是不是确实建在当前目录里

2.6 从工作区走到暂存区

接着执行：

git add notes.txt
git status

这时最关键的变化是：notes.txt 不会再出现在“未追踪文件”那里，而会进入“准备提交”的状态。

不同界面的提示可能不同，但你应该能看到类似：

  • Changes to be committed
  • new file: notes.txt

2.6.1 这一步怎样算成功

算成功：

  • git add 没有报错
  • git status 里能看到 notes.txt 已进入待提交区域
  • 它不再显示为 Untracked

算失败：

  • git add 提示找不到 notes.txt
  • git status 里仍然只有 Untracked files
  • 你在别的目录执行了 git add

如果这一步没成功，通常不是 Git 坏了，而是路径或当前目录不对。

2.7 从暂存区走到历史

继续执行：

git commit -m "添加第一版笔记"
git status

第一次提交时，有些电脑会先提示你设置身份信息，例如用户名和邮箱。如果出现这类提示，可以按它给出的命令先设置。

设置方式通常像这样：

git config --global user.name "你的名字"
git config --global user.email "你的邮箱"

然后再重新执行提交命令。

2.7.1 你现在应该看到什么

提交成功后，Git 通常会输出一小段摘要，告诉你：

  • 这次提交的编号前几位
  • 提交说明
  • 改了几个文件

接着再跑一次 git status，大多数情况下你会看到类似：

nothing to commit, working tree clean

意思是：工作区里没有还没处理的修改，当前状态是干净的。

2.7.2 这一步怎样算成功

算成功：

  • git commit 成功建立一笔提交
  • git status 显示当前没有待提交内容
  • 你已经从“未追踪文件”走到“干净历史”

算失败：

  • git commit 提示没有设置 user.name 或 user.email
  • git commit 提示没有任何内容可提交
  • 你以为提交成功了，但 git status 仍然显示有未处理修改

最后一种情况常见于：你改了文件，但没有先 git add。

2.8 再做一次修改，观察三层重新分开

现在打开 notes.txt，把内容改成：

第一版内容
第二行补充说明

然后执行：

git status
git diff

这时你应该会看到：

  • git status 告诉你文件被修改了
  • git diff 直接显示这次加了哪一行

重点在于：这时修改只在工作区，还没进入暂存区，也没进入历史。

你可以把这三个状态分别想成：

  • 工作区：桌上还在改的稿子
  • 暂存区：已经挑好、准备交出去的部分
  • 历史：已经正式记档的版本

这三个位置一旦在脑子里站稳，后面很多命令都会顺很多。

接着执行：

git add notes.txt
git status
git diff
git diff --staged

这里很值得观察，因为它能帮你真正看见三层。

2.8.1 现在分别在看什么

  • git diff：看工作区和暂存区之间的差异
  • git diff --staged：看暂存区和上一笔提交之间的差异

如果你刚 add 完还没再改文件，常见情况会是：

  • git diff 没内容
  • git diff --staged 能看到你刚才加进去的那一行

这说明修改已经从工作区进入暂存区了。

这里很多人第一次会有一种“原来是这样”的感觉。因为在没看见 git diff 和 git diff --staged 的区别前，工作区和暂存区很容易只是书上的两个词；一旦亲眼看到两条命令输出不同，这两层才会真正立起来。

2.9 为什么理解这件事很重要

很多 Git 初学者会在这里卡很久，因为他们心里其实只有两层：

  • 文件现在长什么样
  • 历史里以前长什么样

但 Git 中间还有一个暂存区。你一旦漏掉这一层，就会觉得很多现象很奇怪，比如：

  • 明明改了文件，为什么提交时说没有内容
  • 明明刚 add 过，为什么 git diff 什么都没显示
  • 为什么有些修改提交了，有些还没有

一旦把三层看清楚，这些现象都会变得合理。

2.10 这一章你至少要练到什么程度

到这里，你不需要已经记住所有命令，但至少应该做到下面这些判断。

如果给你一个文件当前状态，你要能回答：

  • 它现在只是工作区改了，还是已经进暂存区了
  • 它有没有进入历史
  • 下一步该用 git add，还是该用 git commit

如果这三件事你还答不稳，先不要急着往后冲。把这一章的练习多做两遍，比死背命令有效得多。

学 Git 时，最容易让人着急的地方，就是总想赶快进入更像“高级操作”的部分。但如果这一章没站稳，后面的撤销、分支、合并和远端同步都会显得更乱。这个地基打稳，后面会省很多力气。

2.11 本章练习

2.11.1 练习 1：亲手走完三层

在一个测试目录里，从头再做一遍：

  1. git init
  2. 新建一个文件
  3. git status
  4. git add
  5. git status
  6. git commit
  7. 再次修改文件
  8. git diff
  9. git add
  10. git diff --staged

2.11.2 练习 2：判断成功和失败

算成功：

  • 你能说清楚每一步是在处理工作区、暂存区还是历史
  • 你知道为什么 git add 和 git commit 不是一回事
  • 你能根据 git status 的结果判断下一步该做什么

算失败：

  • 你仍然把 git add 理解成“提交”
  • 你看不懂 Untracked、Changes to be committed、modified 这些状态
  • 你不知道 git diff 和 git diff --staged 分别在比什么

如果还没稳，最值得重做的不是整章重读，而是再做一遍最小实验，边做边看状态变化。Git 很多时候不是“看懂”的，而是“看着状态变一次，就慢慢懂了”。
