5  日常修改流程

大多数时候，我们用 Git 不是在做什么复杂操作，而是在重复几个很朴素的动作：改文件、看状态、确认修改、提交。

这一章会把这条最常见的工作流讲顺。你不需要每次都想“接下来该敲什么”，而是让这套顺序变成自然动作。

本章会围绕一个固定节奏展开：

  • 修改文件
  • 用 git status 检查状态
  • 用 git diff 确认内容
  • 用 git add 选择要提交的部分
  • 用 git commit 留下记录

这一章的目标不是增加新命令，而是让你把前面学过的东西真正串成日常习惯。

5.1 为什么要把流程固定下来

很多人学 Git 时，前几次操作都还算顺，一旦开始频繁修改，节奏就容易乱掉。常见情况包括：

  • 改完就急着提交，没先看差异
  • 一口气把一堆不相干的东西一起提交
  • 提交以后才发现漏了一个文件
  • 工作区里一直堆着没处理的改动，自己也搞不清现在到哪一步了

这些问题不是因为 Git 太难，而是因为日常流程没有固定。

最稳的做法很朴素：

  1. 先改
  2. 再看
  3. 再挑
  4. 最后交

只要这四步稳定下来，很多错误会在提交前就被挡住。

5.2 先记住这条最小工作流

平时最常用的节奏，可以先压缩成下面五步：

git status
git diff
git add <文件>
git status
git commit -m "提交说明"

它看起来很简单，但其实把几个关键问题都覆盖了：

  • 我现在改了什么
  • 我要提交哪些东西
  • 还有没有漏掉的文件
  • 这笔提交要怎么命名

5.3 现在做一遍完整日常流程

请在一个已有提交历史的仓库里操作。为了方便练习，下面假设你当前目录里有个 README.md。

先追加一段内容：

printf "\n## 今天的练习\n这一节练习日常提交流程。\n" >> README.md

这一步之后，先不要急着提交。

5.4 第一步：先看状态，不要先猜

执行：

git status

5.4.1 你现在应该看到什么

Git 会告诉你：

  • 哪些文件被修改了
  • 哪些文件还没追踪
  • 有没有已经进暂存区、准备提交的内容

如果你只是改了 README.md，大概率会看到它显示为 modified。

5.4.2 这一步怎样算成功

算成功：

  • 你没有凭感觉判断，而是真的先看了状态
  • git status 的输出和你刚才的操作对得上

算失败：

  • 你其实改了两个文件，却只以为改了一个
  • 状态里已经混进别的旧改动，但你自己没意识到

这也是为什么 git status 几乎是最值得先养成习惯的一条命令。

5.5 第二步：看差异，确认这次到底改了什么

接着执行：

git diff

这一步的重点不是“形式上跑一下”，而是认真确认：

  • 这一段文字是不是我真的想提交
  • 有没有误删内容
  • 有没有顺手改进来的东西其实还不该交

5.5.1 这一步怎样算成功

算成功：

  • git diff 显示的内容和你预期一致
  • 你能明确说出这次改动的目的

算失败：

  • 看到差异以后才发现自己改错地方了
  • 你原本只想改一行，结果 diff 里冒出很多无关内容

如果出现第二种情况，不要硬着头皮提交。先停下来，把无关改动清掉或拆开，再继续。

5.6 第三步：只把这次要交的内容放进暂存区

现在执行：

git add README.md
git status

5.6.1 你现在应该看到什么

README.md 应该进入 Changes to be committed。这表示：你已经明确选中了这次要提交的内容。

如果仓库里还有别的修改，但你没有 add 它们，它们应该仍然留在工作区，不会一起被提交。

5.6.2 这一步怎样算成功

算成功：

  • 只有你想交的文件进入暂存区
  • 你知道这次提交的边界在哪里

算失败：

  • 你把不相干的文件也一起 add 进去了
  • 你以为自己已经选好了，结果状态里还有该提交的文件没进暂存区

所以这一步之后再看一次 git status 很重要，它是在帮你做提交前的最后检查。

5.7 第四步：提交前再检查一次

很多人会觉得：既然已经 git add 了，为什么还要再看一次状态。

原因很简单：你要确认“准备交出去的内容”是不是刚好就是这次该交的内容。

这时你可以再执行：

git status
git diff --staged

5.7.1 你现在应该看到什么

  • git status 会告诉你哪些内容已经准备提交
  • git diff --staged 会直接把“这次准备提交的差异”展开给你看

这一步很像寄信前再看一眼收件人和内容。多花十秒，后面能少很多麻烦。

5.7.2 这一步怎样算成功

算成功：

  • 你确认了这次提交只包含你想交的内容
  • git diff --staged 没出现意料外的改动

算失败：

  • 提交以后才发现多带了别的修改
  • 或者少带了一部分，本来想交的内容其实没进去

5.8 第五步：提交并马上检查结果

确认没问题以后，再执行：

git commit -m "补充日常提交流程示例"
git status
git log --oneline -3

5.8.1 你现在应该看到什么

如果顺利：

  • git commit 会成功建立一笔新提交
  • git status 会回到 working tree clean，或者至少不再显示刚才那部分待提交内容
  • git log --oneline -3 能看到最新那笔提交排在最上面

5.8.2 这一步怎样算成功

算成功：

  • 你完成了一次完整、干净、边界清楚的提交
  • 最新历史和你的实际操作对得上

算失败：

  • 提交说明写得太空，不知道这次交了什么
  • 历史里有新提交，但混进了不相干内容
  • 提交后状态仍然乱成一团

最后一种情况通常说明：你仓库里本来就积着旧改动，没有先整理干净。

5.9 一个常见失败场景：把所有东西一起交掉

这是初学者最容易出现的问题之一。

比如你原本只想提交 README 的说明更新，但仓库里还顺手改了别的文件。结果你图省事，用了：

git add .
git commit -m "update"

表面上这很快，实际上风险很大，因为你可能顺手把下面这些东西也交进去了：

  • 还没写完的内容
  • 临时测试文件
  • 不该进仓库的本地文件
  • 与当前主题无关的改动

5.9.1 这种情况怎样判断成功和失败

算成功：

  • 你知道 git add . 会把当前目录下大量变化一起纳入
  • 你只有在非常确定边界清楚时才这样用

算失败：

  • 你用 git add . 只是因为省事
  • 提交以后才发现混进一堆不该交的东西

对初学者来说，更稳的做法通常是显式写出文件名，先把边界练清楚。

5.10 一个更像真实工作的例子

现在我们模拟一个稍微真实一点的场景。假设你正在写文档，同时做了两件事：

  • 给 README.md 加了一段练习说明
  • 在另一个文件里随手记了几句还没整理好的草稿

这时比较稳的做法不是一把全交，而是：

  1. 先看状态
  2. 先看 diff
  3. 只把已经整理好的那部分 add
  4. 先提交这一部分
  5. 让另一部分继续留在工作区

这样做的好处是，历史会更清楚。以后你看日志时，一眼就知道这笔提交是“补充练习说明”，而不是“一堆事情都顺手改了”。

5.11 日常工作里最值得养成的三个习惯

5.11.1 提交前一定看一次 git diff

不要觉得“我自己刚改的，我当然知道内容”。真正开始长期写作或开发以后，人很容易顺手带进多余修改。git diff 是最便宜的一次自查。

5.11.2 提交后一定看一次 git status

这能帮你确认工作区是否真的回到你预期的状态。很多问题不是出在提交动作本身，而是出在提交完以后还有遗留修改，自己却没注意到。

5.11.3 提交说明尽量只描述这一笔在做什么

不要把说明写成空话，也不要一笔提交塞太多主题。提交越聚焦，后面越省事。

5.12 到这里你应该做到什么

读到这里，你至少应该能稳定跑完这条节奏：

  1. 改文件
  2. git status
  3. git diff
  4. git add
  5. git status
  6. git diff --staged
  7. git commit
  8. git log --oneline

如果这八步你已经能顺着走，后面的撤销、分支和冲突都会容易很多，因为你的基本节奏已经稳了。

5.13 本章练习

5.13.1 练习 1：做一次干净提交

在一个已有仓库里，只修改一个文件，然后完整走一遍：

git status
git diff
git add <文件名>
git status
git diff --staged
git commit -m "..."
git log --oneline -3

算成功：

  • 这笔提交只有一个清楚主题
  • 日志标题和实际改动对得上
  • 提交后状态回到干净

算失败：

  • 你没有检查 diff 就直接提交
  • 提交后才发现多交或漏交

5.13.2 练习 2：故意制造一个“边界不清”的情况

同时改两个文件，但只提交其中一个。

然后观察：

  • git status 会怎么显示
  • 哪个文件进了暂存区
  • 哪个文件还留在工作区

算成功：

  • 你能清楚分辨“已经准备提交”和“还没准备提交”的内容

算失败：

  • 你一旦同时改两个文件，就分不清 Git 现在记录到了哪一步

5.13.3 练习 3：检查自己的习惯

回想你刚才那次提交，问自己：

  1. 我是不是先看了状态
  2. 我是不是先看了 diff
  3. 我是不是在提交前确认过 staged 内容
  4. 我是不是在提交后看了日志

如果这四个问题里有两个以上答不上“是”，说明你的日常节奏还没真正稳定下来。
