9  标签、版本与发布

当一个项目逐渐稳定下来，我们往往会希望把某个时间点明确标出来。比如这是第一版、这是修复 bug 后的版本、这是可以交付给别人的版本。

这一章要讲的，就是标签和版本的作用。标签不是每天都会用到，但一旦项目进入发布节奏，它就会变得很有用。

本章会包括：

  • 为什么需要标签
  • git tag 的基本用法
  • 标签和普通提交有什么不同
  • 怎么用标签整理版本

这一章也会顺手把“版本”这件事讲清楚，让 Git 的历史不只是看得见，还能拿来发布。

9.1 为什么要给某个时间点起名字

前面几章里，我们一直在做提交。提交能把每一次修改记录下来，但当项目慢慢成形以后，你通常还会有另一个需要：

  • 我想把“这个可以交付的状态”明确标出来
  • 我想知道“第一版”对应的是哪一笔提交
  • 我想回到某个发布版本，而不是只记得那是一串提交编号

这时，标签就有用了。

9.2 标签和提交有什么不同

先把最核心的一点说清楚：

  • 提交是在记录修改
  • 标签是在给某个提交起一个更容易识别的名字

也就是说，标签本身通常不承载新改动。它更像一个路标，告诉你：

  • 这里是 v1.0
  • 这里是修完某个问题后的版本
  • 这里是一个适合发给别人用的节点

9.3 什么时候值得打标签

不是每次提交都要打标签。更常见的做法是，在这些时刻考虑打标签：

  • 完成一个明确版本
  • 修完一个重要问题，准备重新交付
  • 项目进入一个稳定节点，准备交付或回看
  • 你想给未来的自己留一个清楚的回看点

如果你的项目还在快速试验阶段，标签可能不需要太频繁。可一旦开始有“版本”意识，标签会非常省事。

9.4 先准备一个练习仓库

这一章最好也在一个已有几笔提交的测试仓库里练。先执行：

git status
git log --oneline -5

9.4.1 这一步怎样算成功

算成功：

  • 当前仓库状态正常
  • 至少已经有几笔提交可供练习

算失败：

  • 你还没在 Git 仓库里
  • 仓库里根本没有历史，没法给某个节点打标签

9.5 第一步：先看现在的历史长什么样

执行：

git log --oneline

9.5.1 你现在应该看到什么

你会看到一串提交记录。现在先不要急着打标签，先看清楚：

  • 哪一笔提交是当前最新的
  • 你想把哪一笔当作“版本节点”

9.5.2 这一步怎样算成功

算成功：

  • 你知道标签最终会落在哪一笔提交上
  • 你不是闭着眼随便打一个标签

算失败：

  • 你还没看清历史，就先乱打标签
  • 打完以后自己都说不清为什么标签会落在那里

9.6 第二步：给当前提交打一个简单标签

如果你想给当前最新提交打上 v1.0，可以执行：

git tag v1.0
git tag

9.6.1 你现在应该看到什么

git tag 会列出当前仓库里的标签。如果这是你第一个标签，通常会看到：

v1.0

9.6.2 这一步怎样算成功

算成功：

  • 仓库里成功出现一个名为 v1.0 的标签
  • 你知道它现在指向当前最新提交

算失败：

  • 你以为标签会自动生成说明，但其实只是给某个提交起了名字
  • 你没确认标签是否真的建成功

9.7 第三步：验证标签到底指到了哪里

只看到标签名字还不够，你最好确认它具体指向哪一笔提交。

执行：

git show v1.0

9.7.1 你现在应该看到什么

Git 会把这个标签对应的提交信息展开给你看。你通常能看到：

  • 这笔提交的说明
  • 它改了什么
  • 它的提交编号

9.7.2 这一步怎样算成功

算成功：

  • 你能确认 v1.0 确实指向你预期的那笔提交
  • 你知道标签不是悬空名字，而是指向具体历史节点

算失败：

  • 你建了标签，却不知道它实际落在哪
  • 你把标签和分支混在一起，不清楚它们的区别

9.8 标签和分支有什么不同

这两个概念很容易混。

  • 分支会继续往前走
  • 标签通常停在某个固定点上

换句话说：

  • 你在分支上继续提交，分支指向会前进
  • 标签打在哪个提交上，通常就固定在那里，不会自己往前移动

这也是为什么标签很适合表示“某一版已经定稿”。

9.9 第四步：继续做一笔提交，看看标签会不会动

现在故意再做一笔新提交：

printf "\n## 版本说明\n这里补一段版本说明。\n" >> README.md
git add README.md
git commit -m "补充版本说明"
git log --oneline -3
git show v1.0

9.9.1 你现在应该看到什么

这时你会看到：

  • 最新历史已经往前多了一笔提交
  • 但 v1.0 仍然指向原来的那个节点

这一步非常重要，因为它能帮你真正看出标签不是“会自动跟着最新状态走”的东西。

9.9.2 这一步怎样算成功

算成功：

  • 你看到新提交出现了
  • 你也看到 v1.0 仍然停在原来的位置

算失败：

  • 你以为打了标签以后，它会永远代表最新状态
  • 你分不清“当前分支最新提交”和“某个固定版本节点”

9.10 第五步：给更早的某笔提交打标签

标签不一定只能打在当前最新提交上。你也可以给历史里的某一笔提交补标签。

先执行：

git log --oneline

找一笔较早的提交，比如：

abc1234 初始化仓库并添加 README

然后执行：

git tag v0.1 abc1234
git tag

9.10.1 你现在应该看到什么

现在标签列表里应该会多一个：

  • v0.1
  • v1.0

其中 v0.1 会指向较早那笔提交，v1.0 指向较新的版本节点。

9.10.2 这一步怎样算成功

算成功：

  • 你能给历史里的指定提交打标签
  • 你知道标签不一定非得打在当前最新提交上

算失败：

  • 你把提交编号写错，结果标签落在错误位置
  • 你没有核对标签对应的提交内容

9.11 标签名该怎么起

标签名没有绝对唯一的标准，但有一些很常见的习惯。

比如：

  • v1.0
  • v1.1
  • v2.0
  • draft-2026-04-19

如果你是在写书或文档，也可以用更贴近自己项目的命名方式，但有两个原则很值得守住：

  • 名字要看得懂
  • 同一项目尽量保持同一套规则

9.11.1 这一步怎样算成功

算成功：

  • 你看到标签名就大致知道它表示什么
  • 项目里的标签命名风格比较一致

算失败：

  • 标签名杂乱无章，后来根本看不懂
  • 同一个项目里一会儿 release-final，一会儿 version2-new，一会儿 ok-this-one

9.12 发布到底在说什么

“发布”这个词有时会让初学者觉得很大，其实先可以理解成一件朴素的事：

  • 把某个你认可的版本明确标出来，准备拿去给别人用、给别人看，或者给未来的自己回头找

所以发布不一定非得是软件上线。对一本书、一个教程、一个文档项目来说，也完全可以有自己的版本节点。

9.13 在常见项目里，标签有什么实际用处

比如你在维护一个个人网站时，可以这样用：

  • 网站骨架搭好后打一个 v0.1
  • 关于页、作品页补齐后打一个 v0.5
  • 第一轮自查并准备上线前打一个 v1.0

这样做的好处是：

  • 你回头找某个阶段很方便
  • 你知道“哪个状态是能拿给别人看的”
  • 你不需要靠模糊印象记“好像上周那版比较稳”

9.14 如果标签打错了怎么办

初学阶段，先知道有这个情况就够了：标签也不是神圣不可碰的。如果你打错了，后面可以删掉重来。

比如：

git tag -d v1.0
git tag

9.14.1 你现在应该看到什么

执行成功后，v1.0 会从本地标签列表里消失。

9.14.2 这一步怎样算成功

算成功：

  • 你知道标签并不是一旦建立就永远不能改
  • 删除后会重新检查列表，而不是靠猜

算失败：

  • 你以为删标签会顺带删掉那笔提交
  • 你把“删标签”和“删历史”混为一谈

标签删掉，不等于那笔提交消失。它只是少了一个名字。

9.15 本章最值得养成的习惯

到这里，这一章最希望你养成的不是“会敲 git tag”，而是这两个习惯：

  • 发布节点要明确标出来
  • 标签打完以后要检查它到底指向哪一笔提交

只要这两件事形成习惯，后面无论是写代码、做网站还是维护文档，版本都会清楚很多。

9.16 本章练习

9.16.1 练习 1：给当前版本打一个标签

  1. git log --oneline
  2. git tag v1.0
  3. git tag
  4. git show v1.0

算成功：

  • 你成功建立标签
  • 你能确认它对应的是哪一笔提交

算失败：

  • 你只建了标签，却没确认它到底指向哪

9.16.2 练习 2：做一笔新提交，观察标签不会自动移动

  1. 修改一个文件并提交
  2. git log --oneline -3
  3. git show v1.0

算成功：

  • 你看见历史往前走了
  • 你也看见旧标签仍然停在原来的节点

算失败：

  • 你以为标签总是代表当前最新状态

9.16.3 练习 3：给旧提交补标签

  1. git log --oneline
  2. 选一笔较早提交
  3. git tag v0.1 <提交编号>
  4. git tag

算成功：

  • 你能把标签打到历史中的某一笔提交上

算失败：

  • 你把标签、分支、提交编号三者混在一起
