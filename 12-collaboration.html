12  用 Pull Request 协作

真正进入团队协作之后，Git 的重点就不再只是“我怎么保存自己的修改”，而是“我怎么把修改有条理地交给别人看，再安全地合并回去”。

这一章会围绕 Pull Request 展开，因为它几乎就是 GitHub 协作的中心动作。你会看到一个改动从分支开始，到提交、推送、发起 PR、接受 review、最后合并的完整过程。

本章重点包括：

  • 为什么协作时通常不直接改主分支
  • Pull Request 在解决什么问题
  • review 时通常会看什么
  • 合并以后本地该怎么整理

这一章读完，GitHub 对你来说就不只是备份网站，而会开始像一个协作平台。

12.1 为什么协作时不直接改主分支

一个人自己写小东西时，偶尔直接在主分支上改，未必马上出事。可一旦进入协作，这样做很快就会出问题：

  • 半成品会直接出现在主线
  • 别人还没来得及看，你的修改就已经进去了
  • 两个人同时改主线，冲突和混乱都会变多
  • 很难区分“正在做的工作”和“已经确认可以合并的工作”

所以更稳的做法通常是：

  • 主分支保持相对稳定
  • 每个任务开一个分支
  • 分支完成以后，用 Pull Request 交给别人看

12.2 Pull Request 在解决什么问题

如果只用一句话来讲，Pull Request 解决的是这件事：

  • 我已经在一个分支上整理好改动了，现在请你看一眼，再决定要不要合并进主线

所以 Pull Request 不是单纯的“发通知”，它实际上把几件事绑在了一起：

  • 展示这次改了什么
  • 给别人评论和 review 的位置
  • 提供合并入口
  • 留下这次协作的记录

12.3 先准备一个最小协作场景

为了练这一章，你至少需要有一个已经连到 GitHub 的仓库。开始前先执行：

git status
git branch
git remote -v

12.3.1 这一步怎样算成功

算成功：

  • 当前仓库状态正常
  • 你知道自己现在站在哪个分支上
  • 这个仓库已经连上 GitHub 远端

算失败：

  • 你还没有远端仓库，却直接跳进 Pull Request
  • 你还没确认当前分支，就开始做协作流程

如果还没连远端，先回到上一章，把 GitHub 对接那一步补上。

12.4 第一步：为一个具体任务建分支

假设你现在要补一节“常见错误”，不要直接在 main 上写。先切一个新分支：

git switch -c add-common-mistakes
git branch

12.4.1 你现在应该看到什么

你通常会看到：

* add-common-mistakes
  main

或者主分支名是 master。

12.4.2 这一步怎样算成功

算成功：

  • 你已经站在新分支上
  • 这次工作和主分支隔开了

算失败：

  • 你以为切过去了，其实还在主分支
  • 你后面所有修改其实都直接做在主线上

12.5 第二步：在分支上完成一笔清楚的修改

现在修改一个文件，比如 README.md 或某个章节文件：

printf "\n## 常见错误\n这里补一节常见错误。\n" >> README.md
git status
git diff
git add README.md
git commit -m "补充常见错误小节"

12.5.1 你现在应该看到什么

这一步结束后，你应该已经在 add-common-mistakes 分支上留下了一笔独立提交。

12.5.2 这一步怎样算成功

算成功：

  • 你在任务分支上完成了一笔主题明确的提交
  • 提交说明和改动内容能对得上

算失败：

  • 你一边改这个主题，一边顺手塞进别的无关修改
  • 提交后自己也说不清这笔提交到底在做什么

12.6 第三步：把任务分支推到远端

现在把这个分支推到 GitHub：

git push -u origin add-common-mistakes

12.6.1 你现在应该看到什么

推送成功后，终端通常会告诉你：

  • 这个新分支已经推到远端
  • 本地分支开始跟踪远端对应分支

很多时候，终端还会给出一个 GitHub 页面链接，方便你直接去发 Pull Request。

12.6.2 这一步怎样算成功

算成功：

  • 远端已经出现 add-common-mistakes 这个分支
  • 本地和远端分支建立了跟踪关系

算失败：

  • 你没确认当前分支名，就把错误分支推上去了
  • 推送失败，却以为网页上已经有这次改动

12.7 第四步：在 GitHub 上发起 Pull Request

现在回到 GitHub 页面。你通常会看到提示，告诉你刚推上来的分支可以发起 Pull Request。

这一步在网页上完成。发 PR 时，通常要确认几件事：

  • base 分支是不是 main 或你的主分支
  • compare 分支是不是 add-common-mistakes
  • 标题能不能让人一眼看懂这次改了什么
  • 描述里有没有交代这次修改的目的

12.7.1 一个最小可用的 PR 标题和描述

比如：

  • 标题：补充常见错误小节
  • 描述：新增一节常见错误，帮助读者更快定位初学阶段的常见问题。

12.7.2 这一步怎样算成功

算成功：

  • PR 指向了正确的主分支
  • 标题和描述清楚说明了改动内容
  • GitHub 页面上能直接看到这次差异

算失败：

  • base 分支选错
  • compare 分支选错
  • PR 标题写得太空，看不出这次在改什么

12.8 第五步：review 时到底在看什么

Pull Request 打开以后，review 不只是“看一眼能不能过”。更稳的检查顺序通常包括：

  • 这次改动的目标清不清楚
  • 改动范围是不是和目标一致
  • 有没有把不相干内容混进来
  • 文本、代码或结构有没有明显问题

对写书、写文档这类项目来说，review 时尤其值得看：

  • 这段话有没有讲清楚
  • 例子顺不顺
  • 有没有和前文重复或矛盾
  • 有没有顺手把别的半成品一起带进来

12.9 第六步：根据 review 调整，再继续推送

如果 review 里有人提了修改意见，不需要重开一个新 PR。通常你只要继续在同一个分支上改，再 push 一次就可以。

比如：

printf "\n补一行说明，回应 review 意见。\n" >> README.md
git add README.md
git commit -m "根据 review 补充说明"
git push

12.9.1 你现在应该看到什么

如果这个分支已经和远端建立过跟踪关系，直接 git push 就可以把新提交送上去。GitHub 上同一个 PR 页面会自动更新。

12.9.2 这一步怎样算成功

算成功：

  • 你在原分支上继续补充修改
  • PR 页面自动出现新的提交和差异

算失败：

  • 你以为 review 后必须重开一个新 PR
  • 你把修改做到别的分支上，结果当前 PR 根本没更新

12.10 第七步：合并 Pull Request

当这次改动已经确认没问题，就可以在 GitHub 上执行合并。

合并以后，通常会发生两件事：

  • 主分支吸收了这个任务分支的工作
  • 这个任务分支可能就可以删掉了

12.10.1 这一步怎样算成功

算成功：

  • 主分支已经包含本次改动
  • PR 在 GitHub 上显示为已合并

算失败：

  • 你以为 PR 开出来就等于已经合并
  • 你没有确认主分支是否真的吸收了这次修改

12.11 第八步：合并后把本地整理干净

很多人做到“网页上点了 merge”就停了，但本地也要跟着整理。

先回到主分支并同步：

git switch main
git pull
git branch

如果主分支叫 master，这里就换成 master。

然后如果你确认本地任务分支已经不再需要，可以删掉：

git branch -d add-common-mistakes
git branch

12.11.1 你现在应该看到什么

整理完成后，理想状态通常是：

  • 你已经回到主分支
  • 主分支已经包含刚才合并进来的工作
  • 本地不再留着没必要的旧任务分支

12.11.2 这一步怎样算成功

算成功：

  • 本地主分支已经同步到最新状态
  • 已完成的任务分支被清理掉

算失败：

  • 远端已经合并了，本地主分支却还停在旧状态
  • 你积了一堆过期分支，自己也分不清哪些还在用

12.12 为什么 Pull Request 对一个人写书也有用

很多人会觉得，PR 好像只是团队开发才需要。其实一个人写书、写文档时，它也有价值。

比如你可以这样用：

  • 每个章节大改开一个分支
  • 写完以后从网页上看一次整段 diff
  • 把 PR 当成一次正式自查入口

这样做的好处是，你会更容易把“正在写”和“准备收进主线”的状态分开。

12.13 协作时最容易犯的几个错

12.13.1 直接在主分支上改

这样会让 review 和回看都变得很乱。

12.13.2 一个 PR 混太多主题

如果一个 PR 同时改结构、改措辞、补案例、修格式，review 会很吃力。

12.13.3 发了 PR 但不检查 base 分支

这会导致改动送到错误目标上。

12.13.4 合并后不整理本地

网页上已经合并了，不代表你本地就自动跟上了。

12.14 本章练习

12.14.1 练习 1：为一个小任务发起 PR

  1. git switch -c <任务分支>
  2. 修改一个文件并提交
  3. git push -u origin <任务分支>
  4. 在 GitHub 页面发起 PR

算成功：

  • 你完成了从分支到 PR 的整条流程
  • PR 标题和修改目标能对得上

算失败：

  • 你没有从分支开始，而是直接在主分支上改

12.14.2 练习 2：在同一个 PR 里继续补修改

  1. 在原分支继续修改
  2. git add
  3. git commit
  4. git push
  5. 刷新 PR 页面

算成功：

  • 原 PR 页面自动更新
  • 你知道 review 后的补改通常还在同一个分支里继续

算失败：

  • 你把修改做到别的地方，导致 PR 没更新

12.14.3 练习 3：合并后整理本地

  1. 在 GitHub 上完成 merge
  2. git switch main
  3. git pull
  4. git branch -d <任务分支>

算成功：

  • 本地主分支已经同步到最新
  • 已完成分支被清理掉

算失败：

  • 远端已经合并，本地却还停在旧状态
