13  Fork 与参与别人的项目

前面讲的是同一个仓库里的协作，这一章会进入另一种常见场景：你没有某个项目的直接写入权限，但还是想参与修改。

这时就会用到 Fork。它让你先把别人的仓库复制到自己的账号下，再从自己的副本开始修改，最后把改动送回原项目。

本章会讲：

  • Fork 适合什么场景
  • Fork 之后本地和上游仓库是什么关系
  • 怎么保持自己的副本不过时
  • 怎么从 Fork 发出一个清楚的贡献

如果你想参与开源项目，这一章会是很实用的起点。

13.1 Fork 适合什么场景

如果你参与的是自己的团队仓库，通常会直接在同一个远端里建分支、发 PR。可如果你没有原仓库的写入权限，情况就不一样了。

这时更常见的流程是：

  • 先 Fork 原项目到自己的 GitHub 账号下
  • 从自己的副本开始改
  • 最后从自己的 Fork 向原项目发 Pull Request

所以 Fork 最适合的，就是“我想参与，但我没有直接写权限”的场景。

13.2 先把三个仓库关系想清楚

这一章最容易乱的地方，不是命令，而是仓库关系。你最好先把这三个角色分清楚：

  • 原项目仓库，也常叫上游仓库
  • 你自己账号下的 Fork 仓库
  • 你电脑里的本地仓库

后面很多命令之所以让人乱，往往不是命令本身难，而是没先想清楚自己现在在和哪一个仓库说话。

13.3 第一步：在 GitHub 上 Fork 一个项目

先在 GitHub 页面上对目标项目执行 Fork。

完成以后，你会得到：

  • 原项目还在原来的组织或账号下
  • 你自己的账号下多出一个同名副本

13.3.1 这一步怎样算成功

算成功：

  • 你已经拥有一个属于自己账号的 Fork 仓库
  • 你知道原项目和 Fork 不是同一个地址

算失败：

  • 你以为 Fork 完以后就直接获得了原项目写权限
  • 你分不清哪个是原项目，哪个是自己的副本

13.4 第二步：把自己的 Fork 克隆到本地

现在应该从你自己的 Fork 地址来 clone，而不是直接从原项目地址 clone。

执行时大致会像这样：

git clone https://github.com/yourname/project.git
cd project
git remote -v

13.4.1 你现在应该看到什么

如果你是从自己的 Fork clone，下来的本地仓库里，默认远端通常会指向你的 Fork：

origin  https://github.com/yourname/project.git (fetch)
origin  https://github.com/yourname/project.git (push)

13.4.2 这一步怎样算成功

算成功：

  • 本地仓库已经从你的 Fork 拉下来
  • 默认远端 origin 指向的是你的 Fork，而不是原项目

算失败：

  • 你 clone 的是原项目地址，却以为那是自己的 Fork
  • 后面一推送才发现自己根本没权限

13.5 第三步：把原项目登记为上游仓库

现在还差一个关键步骤：你要把原项目也登记进本地仓库，通常名字叫 upstream。

执行：

git remote add upstream https://github.com/original-owner/project.git
git remote -v

13.5.1 你现在应该看到什么

理想状态下，你会同时看到两组远端：

origin    https://github.com/yourname/project.git (fetch)
origin    https://github.com/yourname/project.git (push)
upstream  https://github.com/original-owner/project.git (fetch)
upstream  https://github.com/original-owner/project.git (push)

13.5.2 这一步怎样算成功

算成功：

  • origin 指向你的 Fork
  • upstream 指向原项目
  • 你能清楚分辨这两个远端分别是干什么的

算失败：

  • 你把 origin 和 upstream 配反了
  • 你根本没登记上游仓库，后面却想同步原项目变化

13.6 第四步：从自己的分支开始做修改

不要直接在本地主分支上乱改。先为这次贡献开一个分支：

git switch -c fix-typo

然后完成你的修改，再提交：

git add <文件名>
git commit -m "修正文档错字"

13.6.1 这一步怎样算成功

算成功：

  • 你的改动发生在独立任务分支上
  • 提交主题清楚，边界不乱

算失败：

  • 你直接在主分支上改
  • 你一笔提交里混了很多不相干事情

13.7 第五步：把任务分支推到自己的 Fork

现在把这条分支推到你自己的远端：

git push -u origin fix-typo

13.7.1 你现在应该看到什么

推送成功后：

  • 你的 Fork 上会出现 fix-typo 分支
  • 本地分支开始跟踪远端对应分支

13.7.2 这一步怎样算成功

算成功：

  • 这次修改已经出现在你自己的 Fork 上
  • 你不是在试图直接向原项目推送

算失败：

  • 你推的是错误远端
  • 你以为自己在推 Fork，实际上在推原项目地址

13.8 第六步：从自己的 Fork 向原项目发 Pull Request

现在回到 GitHub 页面，在你自己的 Fork 上发起 Pull Request。

这时要特别看清两件事：

  • base 是原项目主分支
  • compare 是你 Fork 里的任务分支

13.8.1 这一步怎样算成功

算成功：

  • PR 的目标是原项目，而不是你自己的 Fork 主分支
  • PR 标题和描述清楚说明了改动内容

算失败：

  • 你把 PR 发回了自己的 Fork
  • 你没看清 base 和 compare，结果方向错了

13.9 第七步：如果原项目更新了，怎么让自己的 Fork 跟上

这是 Fork 工作流里很常见的一步。原项目继续往前走，而你的 Fork 如果不跟，就会慢慢落后。

先取回上游变化：

git fetch upstream

然后回到本地主分支：

git switch main

如果你的默认分支是 master，就换成 master。

然后把上游主分支合进来：

git merge upstream/main

如果上游默认分支是 master，就换成 upstream/master。

最后再把更新后的主分支推回你自己的 Fork：

git push origin main

13.9.1 你现在应该看到什么

完成后，通常会形成这样一种状态：

  • 本地主分支已经包含原项目最新变化
  • 你自己的 Fork 主分支也跟上了上游

13.9.2 这一步怎样算成功

算成功：

  • 你的 Fork 不再落后太多
  • 你知道同步上游时，自己是在和 upstream 打交道

算失败：

  • 你只会从自己的 Fork 拉，结果永远同步不到原项目最新变化
  • 你把同步方向搞反了

13.10 Fork 工作流里最容易混乱的地方

13.10.1 把 origin 和 upstream 混为一谈

这一点最致命。你得始终记住：

  • origin 通常是你自己的 Fork
  • upstream 通常是原项目

13.10.2 从错误的仓库 clone

如果你一开始就 clone 错地址，后面整条线都会乱。

13.10.3 在主分支上长期堆改动

这会让后面同步上游和发 PR 都变得更麻烦。

13.10.4 PR 发错方向

你本来是要贡献给原项目，结果 PR 发回了自己的 Fork。

13.11 为什么参与开源时要尽量把改动做小

当你给别人的项目提贡献时，越小、越单一、越说得清楚的改动，越容易被接受。

比如：

  • 修一个错字
  • 补一条文档说明
  • 修一个明确的小 bug

这些都比“大改一整套结构”更适合作为前几次贡献。

13.12 一个更稳的贡献顺序

如果你是第一次参与别人的项目，比较稳的顺序通常是：

  1. 先 Fork
  2. clone 自己的 Fork
  3. 配上 upstream
  4. 开一个小分支
  5. 做一个小而清楚的改动
  6. 推到自己的 Fork
  7. 再向原项目发 PR

这个顺序不花哨，但很稳。

13.13 本章练习

13.13.1 练习 1：分清三个仓库角色

请用自己的话回答：

  1. 哪个是原项目
  2. 哪个是你的 Fork
  3. 哪个是你本地仓库

算成功：

  • 你能清楚分出这三者

算失败：

  • 你一直把 Fork 和原项目当成同一个仓库

13.13.2 练习 2：配置 origin 和 upstream

  1. git remote -v
  2. git remote add upstream <原项目地址>
  3. git remote -v

算成功：

  • 你能看到 origin 和 upstream 两组地址
  • 你知道谁是谁

算失败：

  • 你把两个远端配反了却没发现

13.13.3 练习 3：完成一次最小贡献流程

  1. 开任务分支
  2. 做一个小修改并提交
  3. git push -u origin <任务分支>
  4. 从 Fork 向原项目发 PR

算成功：

  • 你完成了一条完整的 Fork 贡献路径
  • PR 方向正确，目标是原项目

算失败：

  • 你推送方向或 PR 方向弄错
  • 你还没分清远端关系就急着提交贡献
