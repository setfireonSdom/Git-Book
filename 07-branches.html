7  分支是什么

分支常常是 Git 真正开始变难的地方。不是因为命令特别复杂，而是因为很多人从一开始就没想清楚：分支到底是什么，它为什么能让多人协作变得更顺。

这一章不会把分支讲成抽象名词，而是从一个很直接的问题开始：如果你想改到一半、又不想把半成品放到主线里，该怎么办？

本章会讲清楚：

  • 为什么需要分支
  • 分支不是文件副本，而是一条独立的提交线
  • 怎么建立、切换、查看分支
  • 修改完成后怎么合并回主线

分支一旦理解，Git 就会从“个人备份工具”变成真正能支撑开发流程的工具。

7.1 为什么一直在主线上改会出问题

如果你所有修改都一直堆在主分支上，短期看起来很省事，长期通常会出现这些问题：

  • 改到一半的功能也混在主线里
  • 两件不同主题的工作互相打架
  • 你想试一个想法，却怕把当前稳定版本弄乱
  • 多人协作时，大家都挤在同一条线上改，冲突会明显增多

分支的出现，就是为了让你能把“正在试的东西”和“当前稳定主线”分开。

7.2 分支不是复制出一整个项目

这是最容易误会的地方。

很多初学者第一次听到分支，会以为它像是“复制一个新的文件夹出来”。更准确的理解是：

分支是一条独立向前推进的提交线。

你可以先把它想成：

  • main 是当前默认主线
  • 你从某个提交点上切出去一条新线
  • 后面在这条线上做的新提交，先不会影响主线

也就是说，分支的重点不是“多一个副本”，而是“我可以先在旁边推进一条新历史”。

7.3 先看当前有哪些分支

先在你的测试仓库里执行：

git branch

7.3.1 你现在应该看到什么

如果你现在只有一个默认分支，通常会看到类似：

* main

或者：

* master

前面的 * 表示你当前正站在这个分支上。

7.3.2 这一步怎样算成功

算成功：

  • 你能看懂当前仓库有哪些分支
  • 你能看出自己现在在哪一条分支上

算失败：

  • 你看到输出，却不知道 * 是什么意思
  • 你以为自己在新分支上，其实一直没切出去

7.4 建一个新分支来做新工作

假设你现在想给文档加一节“常见问题”，但还不想马上把半成品放到主线。

可以执行：

git switch -c faq-section
git branch

7.4.1 git switch -c 在做什么

这条命令一次做了两件事：

  • 建立一个新分支 faq-section
  • 立刻切换到这个新分支

7.4.2 你现在应该看到什么

执行完 git branch 以后，通常会看到类似：

* faq-section
  main

这表示：

  • 新分支已经建立成功
  • 你当前正站在 faq-section

7.4.3 这一步怎样算成功

算成功：

  • 新分支真的建立起来了
  • * 已经移动到新分支上

算失败：

  • 你以为自己已经切换，但 * 还在 main
  • 你分支名打错了，后面一直在错误分支上工作

7.5 在新分支上做一次修改

现在在这个新分支上改一个文件。比如给 README.md 补一段：

printf "\n## 常见问题\n这里以后会补常见问题。\n" >> README.md
git status
git add README.md
git commit -m "添加常见问题小节骨架"
git log --oneline -3

7.5.1 你现在应该看到什么

这时你应该能看到：

  • 新提交已经出现在日志里
  • 但这笔提交目前只属于 faq-section 这条分支

7.5.2 这一步怎样算成功

算成功：

  • 你在新分支上完成了一笔独立提交
  • 你知道这笔提交还没并回主线

算失败：

  • 你根本没切到新分支，就直接在主线上做了修改
  • 你以为分支能自动帮你隔离，但实际操作一直还在 main

7.6 切回主分支，观察差异

现在执行：

git switch main
git log --oneline -3

如果你的默认分支叫 master，这里就把 main 换成 master。

7.6.1 你现在应该看到什么

切回主分支以后，你通常会发现：

  • 刚才那笔“添加常见问题小节骨架”的提交不在主线日志顶端
  • README.md 也会回到主分支当前的状态

这就是分支隔离的效果。

7.6.2 这一步怎样算成功

算成功：

  • 你能看到切分支以后，历史视角发生了变化
  • 你理解为什么同一个仓库里，不同分支可以看到不同的最新提交

算失败：

  • 你以为文件应该一直保持刚才那样，结果一切回主线就慌了
  • 你没有理解“当前工作区会跟着当前分支所指向的提交变化”

第一次看到文件内容跟着分支切换而变化，很多人会有点不习惯，但这是正常现象。

7.7 把新分支合并回主线

如果你确认 faq-section 上那笔工作已经准备好了，就可以把它合并回主分支。

先执行：

git switch main
git merge faq-section
git log --oneline -4

7.7.1 git merge 在做什么

它的作用是：把 faq-section 分支上的改动，合并到当前所在分支。因为你现在站在 main 上，所以它会把 faq-section 的工作带回主线。

7.7.2 你现在应该看到什么

如果这次合并比较简单，没有和主线产生冲突，常见情况会是：

  • 合并顺利完成
  • main 上也能看到那笔新工作了
  • README.md 现在包含“常见问题”这一节

7.7.3 这一步怎样算成功

算成功：

  • 你确认自己是在主分支上执行 merge
  • 新分支的修改成功进入主线
  • 日志能反映这次合并后的结果

算失败：

  • 你站错分支，在错误位置执行了 merge
  • 合并后效果和预期不一致，却没有立刻检查日志和文件内容

7.8 合并后怎么检查自己有没有真的做对

合并完成以后，不要只看命令有没有报错。至少检查三件事：

git status
git log --oneline -5
git branch

7.8.1 检查重点

  • git status：当前工作区是不是干净
  • git log --oneline -5：主线最近历史里有没有包含刚才的工作
  • git branch：你现在是不是确实站在预期分支上

7.8.2 这一步怎样算成功

算成功：

  • 主线已经包含新修改
  • 仓库状态正常
  • 你知道自己当前在哪条分支上

算失败：

  • 合并做完后，你其实不知道最后结果长什么样
  • 你只看命令没报错，却没验证历史和文件内容

7.9 为什么分支对单人也有用

很多人会说：我一个人写东西，也需要分支吗？

需要，而且很有用。因为分支不只是为多人准备的，它也在帮你把不同阶段的工作隔开。

例如这些场景都很适合：

  • 你想试一个新写法，但不确定要不要保留
  • 你想重构一段内容，但还不想影响当前稳定版本
  • 你要同时处理“补充新内容”和“修错字”两类事情

没有分支时，你只能把这些东西混在一起。用了分支以后，你就能更从容地试。

7.10 初学者最容易踩的几个坑

7.10.1 以为自己已经切分支了，其实没有

所以每次切完都值得跑一下：

git branch

看看 * 到底在哪。

7.10.2 在主线上做了一堆实验性改动

这会让主线变得很乱，也让后面整理历史更麻烦。

7.10.3 合并后不检查结果

命令成功，不代表内容一定就是你想要的。日志、状态和文件内容都值得看一眼。

7.11 到这里你应该做到什么

这一章结束时，你至少应该能独立完成：

  • 查看当前分支
  • 建立一个新分支并切过去
  • 在新分支上完成修改和提交
  • 切回主分支
  • 把新分支合并回主线

如果这些动作你已经能顺着做，后面理解 Pull Request 和多人协作时会轻松很多，因为它们基本上都是建立在“分支隔离工作，再并回主线”这条思路上。

7.12 本章练习

7.12.1 练习 1：建立并切换分支

  1. git branch
  2. git switch -c test-branch
  3. git branch

算成功：

  • 新分支建立成功
  • * 出现在 test-branch 前面

算失败：

  • 你以为建好了，但其实还站在原分支

7.12.2 练习 2：在新分支上做一笔独立提交

  1. 修改一个文件
  2. git add <文件>
  3. git commit -m "..."
  4. git log --oneline -3

算成功：

  • 你在新分支上留下一笔清楚的提交

算失败：

  • 你没有先确认当前分支，就直接开始改

7.12.3 练习 3：切回主线并合并

  1. git switch main
  2. git merge test-branch
  3. git log --oneline -5

如果默认分支不是 main，请替换成你的实际分支名。

算成功：

  • 主线已经包含 test-branch 的工作
  • 你能从日志看出合并后的结果

算失败：

  • 你切错分支
  • 合并完以后没检查历史和文件
