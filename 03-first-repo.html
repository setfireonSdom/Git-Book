3  第一次建立仓库

理解了基本概念之后，下一步就是亲手建一个仓库，把 Git 真正跑起来。

这一章会从最小流程开始：建立一个目录、初始化 Git、加入文件、查看状态、做出第一笔提交。重点不是把命令一次列满，而是让你亲眼看到 Git 从“还没开始追踪”到“已经留下历史”的过程。

本章会带你完成：

  • 用 git init 建立仓库
  • 用 git status 看懂目前状态
  • 用 git add 把修改放进暂存区
  • 用 git commit 建立第一笔历史
  • 用 .gitignore 排除不该追踪的文件

读完这一章，你应该已经能独立开始一个小项目，并知道 Git 现在记住了什么。

3.1 先把这一章的验收标准放清楚

这一章不是“跟着敲完就算结束”。更实际的标准是：做到最后时，你应该能独立回答下面几件事。

  • 我这个目录现在是不是 Git 仓库
  • 哪些文件已经被追踪，哪些还没有
  • 第一笔历史有没有真的建起来
  • .gitignore 到底在排除什么

如果这四个问题你都能答得稳，这一章才算真的过关。

3.2 这一章要做出什么

我们先定一个很具体的目标。做完这一章时，你手上应该有一个最小 Git 仓库，里面至少有：

  • 一个被 Git 追踪的文本文件
  • 一笔真正提交成功的历史
  • 一个简单的 .gitignore

如果这三样东西都在，你就不是“看过 Git”，而是真的开过一个仓库了。

3.3 练习环境怎么准备

为了避免把练习和你正在做的项目混在一起，先单独建一个测试目录。终端里执行：

mkdir git-first-repo-demo
cd git-first-repo-demo

如果你已经有一个同名目录，也可以换个名字。重点不是目录叫什么，而是你要清楚：接下来的每一步都发生在这个目录里。

3.3.1 这一步怎样算成功

算成功：

  • mkdir 和 cd 都没有报错
  • 终端当前目录已经变成 git-first-repo-demo

算失败：

  • mkdir 提示目录已存在
  • 你执行完 cd 后，其实还停在原来的目录

如果目录已存在，又不确定里面有没有旧内容，最好换一个新的目录名，避免练习结果和旧文件混在一起。

3.4 第一步：初始化仓库

在刚才的目录里执行：

git init

这条命令的意思是：告诉 Git，从现在开始，这个目录要当作一个仓库来管理。

很多人第一次执行 git init 时，会觉得它像是“做了什么很神秘的初始化”。其实你可以先把它想得朴素一点：从这一刻起，Git 开始在这个目录里建立自己的记录方式了。

3.4.1 你现在应该看到什么

通常会看到类似下面的提示：

Initialized empty Git repository in ...

不同系统上的文字可能不完全一样，但核心意思都是一样的：这个目录已经被初始化成一个 Git 仓库。

3.4.2 这一步怎样算成功

算成功：

  • git init 没有报错
  • 当前目录已经变成 Git 仓库

算失败：

  • 出现 git: command not found
  • 你其实不在预期目录里，却把别的目录初始化成了仓库

如果第二种情况发生了，不要急着乱删东西，先用 pwd 看清楚自己当前在哪个目录。

3.5 第二步：建一个文件，让 Git 有东西可管

现在我们建一个简单文件：

printf "# 我的第一个 Git 仓库\n" > README.md
git status

这里的 README.md 只是一个例子。很多项目都会有这个文件，用来说明这个仓库是做什么的。

这里选 README.md 还有一个好处：它足够简单，不会把你的注意力带去业务内容本身。你现在要练的不是“写什么”，而是“Git 怎样开始接管一个文件”。

3.5.1 你现在应该看到什么

git status 大概率会显示：

  • 有一个 README.md
  • 它目前还没有被追踪

你常见到的提示会是：

Untracked files:
  README.md

3.5.2 这一步怎样算成功

算成功：

  • README.md 已经建出来了
  • git status 能看到它
  • 它显示为 Untracked

算失败：

  • 文件没有真的建立成功
  • 你建了文件，但不在当前目录
  • git status 没看到这个文件

如果 git status 看不到它，先执行一次 ls，确认文件是不是真的在当前目录里。

3.6 第三步：把文件加入暂存区

现在执行：

git add README.md
git status

这一步不是“已经提交”，只是把 README.md 放进本次准备提交的清单。

3.6.1 你现在应该看到什么

这时 README.md 应该从“未追踪文件”移动到“准备提交的修改”。

常见提示像这样：

Changes to be committed:
  new file: README.md

3.6.2 这一步怎样算成功

算成功：

  • git add 没报错
  • README.md 不再出现在 Untracked files
  • 它出现在 Changes to be committed

算失败：

  • git add 报错，提示找不到文件
  • git status 里仍然显示它未被追踪

这种失败一般不是 Git 有问题，而是文件名、路径或当前目录不对。

3.7 第四步：提交第一笔历史

现在执行：

git commit -m "初始化仓库并添加 README"
git status

3.7.1 如果 Git 要你先设置身份

第一次在这台电脑上提交时，Git 可能会要求你先设置用户名和邮箱。如果看到类似提示，可以先执行：

git config --global user.name "你的名字"
git config --global user.email "你的邮箱"

然后再重新执行提交命令。

3.7.2 你现在应该看到什么

提交成功后，Git 通常会输出一段简短信息，里面会包含：

  • 本次提交的简短编号
  • 你写的提交说明
  • 这次改动涉及几个文件

随后再看 git status，你应该会看到：

nothing to commit, working tree clean

这句话很重要，它表示当前工作区和暂存区都没有还没处理的内容。

3.7.3 这一步怎样算成功

算成功：

  • 第一笔提交已经建立起来
  • git status 显示工作区是干净的
  • 你已经有了一个最小可用仓库

算失败：

  • git commit 因为身份信息没设好而失败
  • git commit 提示没有东西可提交
  • 提交后 git status 仍然显示有文件没处理

第二种情况通常说明：你忘了先 git add。

这也是初学者最容易混淆的一点：改了文件不等于已经进入历史。Git 不会替你脑补“你大概是想提交吧”。你得明确告诉它，这次要交哪些内容。

3.8 第五步：确认历史真的存在

很多初学者做到这里，会以为“状态干净了就算结束”。更关键的是，你要确认历史真的留下来了。

执行：

git log --oneline

3.8.1 你现在应该看到什么

你应该会看到至少一行输出，大致像这样：

abc1234 初始化仓库并添加 README

前面那串短编号会因每台电脑不同而不同，但你应该能看到：

  • 一笔提交记录
  • 你刚才写的提交说明

3.8.2 这一步怎样算成功

算成功：

  • git log --oneline 能显示出刚才那笔提交
  • 你能确认第一笔历史已经存在

算失败：

  • git log 没有输出你预期的提交
  • 你以为提交成功了，其实刚才那一步根本没成功

如果这里没有看到记录，就回头检查上一节的 git commit 是否真的执行成功。

把 git log --oneline 养成习惯很值得。它不是多做一步，而是在帮你确认：我以为自己已经完成的事情，是不是真的已经进入历史。

3.9 第六步：再改一次，感受日常节奏

现在继续修改 README.md，追加一行说明：

printf "\n这是一个用来练习 Git 的测试仓库。\n" >> README.md
git status
git diff

这时你应该看到：

  • README.md 被标记为已修改
  • git diff 显示你新加的那一行

然后执行：

git add README.md
git commit -m "补充仓库说明"
git log --oneline

3.9.1 这一步怎样算成功

算成功：

  • 你又完成了一次完整流程：修改、查看、暂存、提交
  • git log --oneline 里现在至少有两笔提交
  • 最新一笔是 补充仓库说明

算失败：

  • git diff 看不到你预期的改动
  • git commit 后日志里还是只有一笔提交
  • 你把内容改了，但忘了保存文件

最后一种情况很常见，尤其是用编辑器手动修改时。终端命令没问题，不代表文件已经真的保存到了磁盘上。

3.10 .gitignore 是干什么的

到这里，仓库已经能正常工作了。接下来要补一个很实用的东西：.gitignore。

不是所有文件都该交给 Git 管。比如：

  • 临时文件
  • 编辑器自动生成的缓存
  • 编译产物
  • 本地环境文件

这一点越早建立越省事。很多仓库之所以一开始就显得乱，不是因为 Git 难，而是因为不该进仓库的东西太早混进来了。

如果这些文件混进仓库，状态会变得很乱，协作时也容易带来麻烦。

现在执行：

printf ".DS_Store\n*.log\n" > .gitignore
git status

3.10.1 你现在应该看到什么

这时 Git 会把 .gitignore 当作一个新文件看待，因为你刚刚新建了它。它通常会显示在未追踪文件里。

如果你目录里恰好有 .DS_Store 或某个 .log 文件，Git 之后就会忽略它们，不再把它们列为待处理内容。

3.10.2 这一步怎样算成功

算成功：

  • .gitignore 文件建立成功
  • git status 能看到 .gitignore
  • 以后匹配规则的文件不会再反复冒出来

算失败：

  • 你写错了文件名，比如写成 gitignore
  • 忽略规则没有生效，却以为 Git 出了问题

要注意一件事：如果某个文件之前已经被 Git 追踪了，后来才写进 .gitignore，它不会自动消失。.gitignore 主要影响“还没被追踪的文件”。

这一句很值得记住，因为很多人第一次用 .gitignore 时，都会误以为“写进去以后 Git 就会自动把旧文件忘掉”。其实不是这样，它更像是在说：从现在起，这类新文件别再来打扰我。

3.11 把 .gitignore 也提交进去

既然 .gitignore 是仓库规则的一部分，它本身通常也应该提交进历史。

执行：

git add .gitignore
git commit -m "添加基础忽略规则"
git log --oneline

3.11.1 这一步怎样算成功

算成功：

  • .gitignore 也进入历史
  • 日志里能看到第三笔提交

算失败：

  • 你建了 .gitignore，但没提交进去
  • 之后换一台电脑或换一个协作者，这份规则就跟着丢了

3.12 到这里你应该已经具备什么能力

如果前面的流程你都亲手做过一遍，到这一章结束时，你应该已经能独立完成下面这些动作：

  • 建一个新目录并初始化 Git
  • 用 git status 判断当前状态
  • 用 git add 把文件放进暂存区
  • 用 git commit 建立正式历史
  • 用 git log --oneline 检查历史
  • 用 .gitignore 排除一部分不该追踪的文件

这就是一个最小但完整的 Git 起点。

从这一章往后，你会不断往这个最小起点上加东西：看差异、读历史、撤销、分支、远端、协作。可无论后面学多少内容，起点一直都是这里这几步。

3.13 本章练习

3.13.1 练习 1：从零再做一遍

换一个新的测试目录，从头完整做一次，不要看上面的命令逐条照抄，只在卡住时回头检查。

如果你能自己完成下面这些动作，说明这一章已经基本稳了：

  • 建仓库
  • 建文件
  • 做第一笔提交
  • 再改一次并提交
  • 加入 .gitignore

3.13.2 练习 2：判断成功和失败

算成功：

  • 你已经有一个真的 Git 仓库，不是空目录
  • git log --oneline 至少能看到两到三笔提交
  • git status 大多数时候能回到 working tree clean
  • 你知道 .gitignore 是拿来排除什么的

算失败：

  • 你只会照抄命令，但看不懂 git status
  • 你以为 git add 就等于完成提交
  • 你提交过了，却不会检查日志
  • 你不知道仓库里哪些文件该交给 Git，哪些不该

如果还没稳，不要急着往分支和远端走。先把这一章练到顺，因为后面所有操作都建立在这个最小流程上。
