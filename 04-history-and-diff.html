4  怎么看变化，怎么看历史

会提交只是开始。真正让 Git 有价值的，是你之后能看见改了什么、什么时候改的、是谁改的，以及这次修改和上一次到底差在哪里。

这一章会带你学会两件事：怎么看当前变化，怎么看过去历史。前者帮你决定这次要不要提交，后者帮你理解一个项目是怎么一步步变成今天这个样子的。

本章会涵盖：

  • 用 git diff 查看修改内容
  • 用 git log 阅读提交历史
  • 怎么写比较像样的提交信息
  • 什么样的提交以后最容易回头看

如果这一章掌握了，你会开始感受到 Git 不只是“保存一下”，而是在帮你整理整个修改过程。

4.1 为什么这一步很重要

很多初学者学 Git 时，容易把重点放在“怎么提交”。可一旦提交次数变多，真正决定你会不会乱掉的，反而是另外两个能力：

  • 你能不能在提交前看懂自己改了什么
  • 你能不能在几天以后看懂当时为什么那么改

如果没有这两个能力，提交就会慢慢退化成机械动作。表面上你在用 Git，实际上历史还是一团乱。

4.2 先准备一个练习仓库

如果你前一章已经做过 git-first-repo-demo，可以继续沿用那个目录。没有的话，就先建立一个最小仓库，并至少做过一笔提交。

你需要先确保下面这件事成立：

git status
git log --oneline

4.2.1 这一步怎样算成功

算成功：

  • git status 能正常执行
  • git log --oneline 至少能看到一笔提交

算失败：

  • git status 提示当前目录不是 Git 仓库
  • git log --oneline 没有任何提交记录

如果这里还没准备好，就先回到上一章，把仓库初始化和第一次提交补齐。

4.3 第一部分：怎么看当前变化

先改一个文件。假设你当前目录里有 README.md，现在追加两行：

printf "\n## 练习目标\n学会查看差异和历史。\n" >> README.md
git status

4.3.1 你现在应该看到什么

这时 git status 应该会告诉你：README.md 被修改了，但还没有进入暂存区。

常见提示像这样：

modified: README.md

4.3.2 这一步怎样算成功

算成功：

  • 你已经真的改到了文件
  • git status 把 README.md 标成 modified

算失败：

  • 你以为自己改了，但文件其实没保存
  • git status 没看到任何变化

如果状态没有变化，先检查文件是不是改在当前仓库里那一份，而不是改到了别的目录里的同名文件。

4.4 用 git diff 看改了什么

现在执行：

git diff

这条命令会直接比较“工作区现在的内容”和“暂存区里当前准备提交的内容”。因为你现在还没 git add，所以它会把刚才新加的两行显示出来。

4.4.1 你现在应该看到什么

你通常会看到一段带 + 和 - 的输出：

  • 以 + 开头的行，表示新增内容
  • 以 - 开头的行，表示删除内容

如果你刚才只是追加了两行，大概率会看到两行以 + 开头的文本。

4.4.2 这一步怎样算成功

算成功：

  • git diff 能清楚显示你刚才改动的内容
  • 你能看出哪些行是新增，哪些行是删除

算失败：

  • git diff 什么都没显示，但你明明改了文件
  • 你看到输出，却完全分不清哪些是旧内容，哪些是新内容

第一种失败最常见的原因是：你已经先 git add 了。这样工作区和暂存区一致，git diff 就可能没有输出。

4.5 为什么 git diff 有时没输出

这是初学者最容易困惑的地方之一。不是没有改动，而是你比较的两层刚好一样。

比如你执行：

git add README.md
git diff
git diff --staged

这时常见情况会变成：

  • git diff 没输出
  • git diff --staged 有输出

原因不是 Git 出问题了，而是因为：

  • 工作区和暂存区现在一样
  • 暂存区和上一次提交不一样

4.5.1 这一步怎样算成功

算成功：

  • 你能理解为什么 git diff 和 git diff --staged 的结果不同
  • 你知道该用哪条命令看哪一层的差异

算失败：

  • 你把“没输出”理解成“改动消失了”
  • 你不知道修改现在到底停在哪一层

4.6 第二部分：怎么看历史

当前改动看明白以后，下一步就是看历史。先把刚才的改动提交进去：

git commit -m "补充练习目标说明"
git log

4.6.1 你现在应该看到什么

git log 通常会显示一串提交记录，最新的在最上面。每一笔提交一般都会包含：

  • 提交编号
  • 作者
  • 日期
  • 提交说明

如果你觉得这信息太长，也可以用更紧凑的版本：

git log --oneline

这样每笔提交会压成一行，更适合快速浏览。

4.6.2 这一步怎样算成功

算成功：

  • git log 能看到刚才那笔新提交
  • git log --oneline 能按顺序列出历史

算失败：

  • 你刚提交完，但日志里没有新记录
  • 你其实没有提交成功，却误以为已经进入历史

如果日志没有更新，先不要猜，直接重新看上一条 git commit 的输出有没有报错。

4.7 git log 在解决什么问题

它不只是让你“看看过去发生过什么”，更是在帮你回答几个很实际的问题：

  • 这个项目最近改了什么
  • 这段改动是谁加进去的
  • 哪一次提交引入了某个变化
  • 如果我要回头找一版内容，该往哪里看

所以 git log 不是一个可有可无的命令，它是你读项目历史的入口。

4.8 第三部分：怎么写比较像样的提交信息

很多人刚开始写提交信息时，会写成：

  • update
  • fix
  • change file
  • again

这些信息不是完全不能用，但问题是：三天以后你几乎看不出当时做了什么。

比较像样的提交说明，至少应该做到一件事：让未来的你一眼知道这笔提交在解决什么。

例如下面这些就更清楚：

  • 添加 README 初稿
  • 补充练习目标说明
  • 修正文档中的命令示例
  • 添加基础忽略规则

4.8.1 什么样的提交信息算成功

算成功：

  • 只看提交说明，大致就知道这次做了什么
  • 说明和实际改动能对得上

算失败：

  • 提交说明空泛到几乎没有信息
  • 一条说明里混了太多不相干的事情

比如你同时修了错字、改了结构、加了新例子，却只写一个 update，以后就很难回头查。

4.9 一个最小对比实验

你可以现在自己做个实验，连续做两笔提交。

第一笔故意写差一点：

printf "\n临时补一句话。\n" >> README.md
git add README.md
git commit -m "update"

第二笔写清楚一点：

printf "\n再补一行，说明这一章在练习 diff。\n" >> README.md
git add README.md
git commit -m "补充 diff 练习说明"
git log --oneline

4.9.1 你现在应该看到什么

日志里这两笔提交都会存在，但可读性差别会很明显。

当你几分钟后回头看：

  • update 很难帮助你回忆当时做了什么
  • 补充 diff 练习说明 一眼就知道这笔提交在干嘛

这就是提交信息质量的差别。

4.10 第四部分：怎么看一笔提交具体改了什么

光知道“有一笔提交”还不够，你还需要能深入看某一笔到底动了什么。

先执行：

git log --oneline

找一笔你刚做的提交编号，假设是：

abc1234 补充 diff 练习说明

然后执行：

git show abc1234

4.10.1 你现在应该看到什么

git show 会把这笔提交的说明和具体差异一起展开。你通常能看到：

  • 这笔提交的标题
  • 修改了哪个文件
  • 增加了哪些行，删除了哪些行

4.10.2 这一步怎样算成功

算成功：

  • 你能从 git show 里读出这笔提交到底改了什么
  • 你知道怎么从日志跳到具体改动

算失败：

  • 你只会看日志标题，不会检查具体内容
  • 你看见提交编号，却不知道怎么追进去看细节

4.11 到这里你应该形成什么习惯

这一章最想帮你养成的，不是多记几条命令，而是一个很实际的习惯：

提交之前先看差异，提交之后会看历史。

这两步一旦养成，很多混乱会自然减少。因为你不再是“改完就交”，而是会先确认自己到底要交什么；交完以后，也能马上检查历史是否长成你预期的样子。

4.12 本章练习

4.12.1 练习 1：看工作区和暂存区的差异

在一个已有仓库里做下面的事：

  1. 修改一个文件
  2. 运行 git diff
  3. 执行 git add
  4. 再次运行 git diff
  5. 运行 git diff --staged

算成功：

  • 你能解释为什么三次看到的结果不同
  • 你知道修改此刻在工作区、暂存区还是历史

算失败：

  • 你只会机械照抄命令，却说不清每条命令在比较哪两层

4.12.2 练习 2：检查一笔历史

做一笔新提交，然后执行：

git log --oneline
git show <提交编号>

算成功：

  • 你能从日志里找到刚才那笔提交
  • 你能从 git show 看出它具体动了什么

算失败：

  • 你知道有提交，但不会定位到某一笔具体差异

4.12.3 练习 3：判断提交说明写得好不好

回头看你自己最近三笔提交，问自己两个问题：

  1. 只看标题，我知不知道它们分别在做什么
  2. 如果三天后回来，我还能不能靠这些标题快速定位问题

如果答案大多是否定的，就说明提交说明还需要继续练。

算成功：

  • 你只看提交说明，大致就知道每一笔在处理什么
  • 你能分辨哪些说明具体，哪些说明只是空泛地写了 update 或 fix

算失败：

  • 你回头看自己的提交历史，还是只能靠猜
  • 你知道提交做过什么，但标题本身几乎帮不上你
