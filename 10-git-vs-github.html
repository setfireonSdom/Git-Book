10  Git 和 GitHub 不是一回事

很多人把 Git 和 GitHub 当成同一件事，所以一开始就会混乱。其实 Git 是版本控制工具，主要在本地工作；GitHub 是托管远端仓库、支撑协作的平台。两者关系很紧，但不是同一个东西。

这一章的任务，就是把这层关系理清楚。你要知道哪些动作发生在自己电脑上，哪些动作是在和远端同步，哪些事情又是 GitHub 提供的协作功能。

本章会讲：

  • Git 和 GitHub 各自负责什么
  • 什么是远端仓库
  • 本地和远端之间是什么关系
  • 为什么会有 push、pull、fetch

把这一层分清楚以后，后面接 GitHub 时会顺很多。

10.1 先把两个名字拆开

如果只用一句话来区分：

  • Git 是你本地用来记录历史的工具
  • GitHub 是用来托管远端仓库和做协作的平台

这两个东西经常一起出现，所以很容易被混成一个概念。但它们处理的问题并不相同。

10.2 哪些事情是 Git 在做

下面这些动作，就算完全不上 GitHub，你也照样可以在自己电脑上完成：

  • git init
  • git add
  • git commit
  • git diff
  • git log
  • git branch
  • git merge

也就是说，前面几章你学的那些核心操作，本质上都属于 Git 本地能力。哪怕你断网、哪怕你还没注册 GitHub 账号，Git 仍然能工作。

10.3 哪些事情是 GitHub 在做

GitHub 主要处理的是另外一类事情：

  • 托管远端仓库
  • 让别人也能访问这个仓库
  • 提供 Pull Request、Issue、Code Review 这些协作功能
  • 让你的项目放到线上，方便同步和分享

所以更准确的理解不是“GitHub 就是 Git”，而是：

  • Git 是底层版本控制工具
  • GitHub 是围绕 Git 仓库搭建出来的协作平台

10.4 一个最常见的误会

很多初学者会说：“我还没上传到 GitHub，所以还没开始用 Git。”

这句话其实不对。只要你已经在本地做了这些事：

  • 建仓库
  • 提交历史
  • 切分支
  • 查看差异

你就已经在用 Git 了。只是你还没有把这个本地仓库接到 GitHub 而已。

10.5 什么叫远端仓库

当你只在自己电脑上工作时，你手里的是一个本地仓库。它的历史、分支和文件状态都保存在你当前这台机器上。

所谓远端仓库，可以先简单理解成：

  • 放在别处的一份仓库副本
  • 你可以和它同步历史
  • 别人也可能从那里拿到同一个项目

GitHub 上的仓库，最常见的就是这种远端仓库。

10.6 本地仓库和远端仓库是什么关系

它们不是“一个真，一个假”，也不是“本地只是临时，远端才算正式”。

更准确地说，它们是：

  • 两个可以彼此同步的仓库
  • 本地负责你正在做的改动
  • 远端负责共享、备份和协作入口

所以最稳的理解是：你平时主要在本地做 Git 操作，等到合适的时候，再把本地历史同步到远端。

10.7 现在用一个最小例子来想象

假设你在自己电脑上已经有一个仓库，里面做了三笔提交。

这时：

  • git log 能看到三笔历史
  • 但 GitHub 上还什么都没有

一旦你在 GitHub 新建仓库，并把本地仓库连过去：

  • 你的本地仓库还是原来那个
  • GitHub 上会多出一个远端副本

以后你们之间就会通过“同步”来保持更新。

10.8 push 在做什么

push 的意思可以先理解成：

  • 把本地已经存在的提交，送到远端仓库去

注意这句话的重点是：push 发送的是“已经提交过的历史”，不是你工作区里还没提交的半成品。

所以如果你只改了文件，但还没 git commit，直接 push 往往不会得到你以为的结果，因为你其实没有新的历史可推。

10.8.1 这一步怎样算理解成功

算成功：

  • 你知道 push 不是上传当前目录所有现状
  • 你知道它推送的是已经提交的历史

算失败：

  • 你把 push 理解成“保存一下现在电脑上的东西”
  • 你以为没 commit 也能靠 push 把未整理好的修改传上去

10.9 pull 在做什么

pull 可以先理解成：

  • 把远端的新变化拿下来，并尝试合并到本地

也就是说，它通常不是单纯“下载”，而是“取回并整合”。

这也是为什么 pull 有时会带来合并，甚至可能带来冲突。

10.9.1 这一步怎样算理解成功

算成功：

  • 你知道 pull 可能会改变你本地当前分支的状态
  • 你知道它不只是“看一下远端”

算失败：

  • 你把 pull 想成纯粹只读动作
  • 你不知道为什么 pull 之后本地内容会变化

10.10 fetch 又是什么

fetch 和 pull 经常一起被提到，但它更克制一些。

你可以先把它理解成：

  • 先把远端的新历史拿到本地
  • 但先不自动合并进你当前正在工作的分支

所以很多人会把它看成更安全的第一步：先取回来看看，再决定怎么合。

10.10.1 fetch 和 pull 的差别

  • fetch：先拿回来，不自动并进当前工作
  • pull：拿回来，并尝试合进当前工作

10.10.2 这一步怎样算理解成功

算成功：

  • 你知道 fetch 更像“先同步信息”
  • 你知道 pull 比 fetch 多了一步整合

算失败：

  • 你把两者完全当成同一件事
  • 你不知道为什么有些人习惯先 fetch 再决定下一步

10.11 clone 又属于哪一类

虽然这一章还没正式进入 GitHub 实操，但可以先顺手把 clone 放进图里。

clone 的意思是：

  • 根据一个现成远端仓库，复制出一份本地仓库

也就是说：

  • init 常用于从零开始建本地仓库
  • clone 常用于从已有远端仓库拿下一份本地副本

这两个起点不同，别混在一起。

10.12 现在做一个理解检查

先不要急着接 GitHub 网站界面，先问自己下面几个问题：

  1. 如果我断网了，还能不能 git commit
  2. 如果我只改了文件还没提交，push 能不能把这些改动送上去
  3. 为什么 pull 有时会带来冲突
  4. 为什么有人先 fetch 再手动决定下一步

如果这四个问题你已经能答稳，Git 和 GitHub 的边界就开始清楚了。

10.13 一个最小命令观察实验

如果你现在已经有一个连到远端的仓库，可以做一个很小的观察：

git remote -v

10.13.1 你现在应该看到什么

如果这个仓库已经配置过远端，通常会看到类似：

origin  https://github.com/...
origin  https://github.com/...

这里的 origin 是远端名字，后面是远端地址。

10.13.2 这一步怎样算成功

算成功：

  • 你知道当前仓库有没有配置远端
  • 你知道远端不一定非叫 origin，但 origin 很常见

算失败：

  • 你看到远端地址，却不知道它代表什么
  • 你把远端名字和分支名字混为一谈

如果这里没有任何输出，也不代表仓库有问题，只说明这个本地仓库还没连到远端。

10.14 初学者最容易混乱的几个点

10.14.1 把 Git 和 GitHub 当成同义词

这样一来，你会误以为“没上 GitHub 就等于没在用 Git”。

10.14.2 把 push 理解成“上传当前文件夹”

实际上，它推的是提交历史，不是未提交的工作区现状。

10.14.3 把 pull 理解成纯下载

实际上，它往往还会试着整合远端变化。

10.14.4 忽略本地仓库本身的重要性

很多人把 GitHub 想得太重，结果反而忽略了真正每天在你手边工作的，是本地 Git 仓库。

10.15 到这里你应该做到什么

这一章结束时，你至少应该能稳定区分：

  • 哪些操作属于本地 Git
  • 哪些功能属于 GitHub
  • 本地仓库和远端仓库分别在做什么
  • push、pull、fetch 的大致角色差别

一旦这层分清楚，后面真正接 GitHub 时，你就不会把网站按钮和本地历史搅在一起。

10.16 本章练习

10.16.1 练习 1：观察远端信息是不是已经配好

如果你手边已经有一个连到 GitHub 的仓库，先执行：

git remote -v

然后回答两件事：

  1. 当前仓库有没有远端
  2. 这个远端地址是不是你预期的那个仓库

算成功：

  • 你能从输出里认出远端名字和远端地址
  • 你知道自己现在面对的是本地仓库，还是“本地加远端”的仓库

算失败：

  • 你看到 origin 和地址，却不知道它们分别代表什么
  • 你仓库其实没连远端，却误以为已经和 GitHub 建立同步关系

10.16.2 练习 2：只用自己的话区分 Git 和 GitHub

试着不用书里的原句，自己回答：

  1. Git 是什么
  2. GitHub 是什么
  3. 它们为什么常常一起出现

算成功：

  • 你能清楚说出“工具”和“平台”的区别

算失败：

  • 你还是把它们说成同一个东西

10.16.3 练习 3：判断下面几件事是谁在做

请判断这些动作更属于 Git 还是 GitHub：

  • 本地做一笔提交
  • 建立 Pull Request
  • 查看 git diff
  • 在网页上 review 别人的改动
  • 切换分支

算成功：

  • 你能把本地版本控制和平台协作功能分开

算失败：

  • 你看到和项目协作有关，就统统算成 GitHub

10.16.4 练习 4：做一次最小观察，区分 fetch 和 pull

如果你手边已经有一个连到远端的测试仓库，可以做下面这组最小观察：

  1. 先执行 git status
  2. 再执行 git fetch
  3. 再看一次 git status
  4. 最后执行 git branch -r

这组动作的重点不是“做出新改动”，而是观察：

  • fetch 之后，本地当前文件通常不会立刻变化
  • 但 Git 已经把远端分支信息取回来了

如果之后你再执行 git pull，它就不只是“拿回来看看”，而是会尝试把远端变化整合进当前分支。

算成功：

  • 你知道 fetch 之后为什么工作区通常不会立刻变
  • 你知道 pull 比 fetch 多了一步整合
  • 你不会再把两者都看成“差不多的下载命令”

算失败：

  • 你执行完 fetch，发现文件没变，就以为命令没生效
  • 你仍然说不清 fetch 和 pull 的差别到底落在哪一步

10.16.5 练习 5：判断 push、pull、fetch

试着回答：

  1. 哪一个是在把本地历史送到远端
  2. 哪一个是在取回远端变化并尝试整合
  3. 哪一个是先拿回来但先不自动并进当前工作

算成功：

  • 你能分清三者的大致职责

算失败：

  • 你仍然把它们当成三个差不多的“同步命令”
