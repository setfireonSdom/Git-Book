8  冲突怎么解决

冲突几乎是每个 Git 初学者都会怕的一关。屏幕上突然出现一堆标记，看起来像哪里坏掉了，于是很多人第一反应是紧张，甚至想直接关掉。

冲突并不神秘。它只是 Git 在提醒你：同一个地方同时出现了两种修改，现在需要你来决定留下哪一种，或者怎么把两边合起来。

这一章会一步一步拆开这个过程：

  • 冲突通常是怎么产生的
  • 冲突标记各代表什么
  • 解决冲突以后为什么还要再提交
  • 怎么减少无谓的冲突

只要你第一次在讲解下把冲突走完一遍，之后再看到它，心里就会稳很多。

8.1 冲突到底是什么

先把最重要的一句话放在前面：冲突不是 Git 坏了，也不是你的仓库炸了。

冲突的意思只是：

  • 两边都改了同一个位置
  • Git 没法替你判断最后该保留哪一种

如果两个人改的是不同位置，Git 通常能自动合并。真正麻烦的，是你们碰到了同一段内容，而且改法互不兼容。

8.2 为什么会有冲突

冲突最常见于两种场景：

  • 你在分支 A 改了一段文字，主分支后来也改了同一段
  • 两个人分别在不同分支上改了同一文件里的同一区域

所以冲突不是“只要合并就一定会发生”，而是“同一位置有竞争修改时才会发生”。

8.3 先自己制造一次最小冲突

这一章最值得做的，不是只看说明，而是亲手走一遍。第一次自己在练习仓库里制造冲突，后面在真实项目里遇到时，心里会稳很多。

请在一个测试仓库里操作，先确保当前在主分支，并且工作区是干净的：

git status
git branch

8.3.1 这一步怎样算成功

算成功：

  • git status 显示当前没有待处理修改
  • 你知道自己现在站在哪个分支上

算失败：

  • 仓库里本来就堆着旧改动
  • 你还没确认当前分支，就直接开始做冲突实验

如果工作区不干净，建议先处理掉旧改动，再开始。不然待会儿很难分清哪些变化是冲突实验带来的。

8.4 第一步：先在主分支准备一个基础文件

先建立一个文件，并做一笔提交：

printf "第一行：Git 可以帮助我们管理修改。\n第二行：这一段稍后会被改动。\n" > conflict-demo.txt
git add conflict-demo.txt
git commit -m "添加冲突练习文件"

8.4.1 这一步怎样算成功

算成功：

  • 仓库里已经有一个 conflict-demo.txt
  • 这份文件已经提交进历史

算失败：

  • 文件只建出来了，但还没提交
  • 你后面切分支时，其实没有一个共同起点

8.5 第二步：从主分支切出一个新分支

执行：

git switch -c conflict-branch

然后把第二行改成：

第二行：这是分支版本的改动。

可以直接执行：

printf "第一行：Git 可以帮助我们管理修改。\n第二行：这是分支版本的改动。\n" > conflict-demo.txt
git add conflict-demo.txt
git commit -m "在分支上修改第二行"

8.5.1 这一步怎样算成功

算成功：

  • conflict-branch 已经建立成功
  • 你在这个分支上完成了一笔提交

算失败：

  • 你以为自己在新分支上，实际上还在主分支
  • 你修改了文件，但忘了提交

8.6 第三步：切回主分支，改同一行

现在回到主分支：

git switch main

如果你的默认分支是 master，这里就改成 master。

然后把同一行改成另一个版本：

printf "第一行：Git 可以帮助我们管理修改。\n第二行：这是主分支版本的改动。\n" > conflict-demo.txt
git add conflict-demo.txt
git commit -m "在主分支上修改第二行"

8.6.1 这一步怎样算成功

算成功：

  • 主分支上也完成了一笔对同一行的修改
  • 两个分支现在各自有不同版本

算失败：

  • 你改的不是同一行，结果后面不会产生真正的冲突
  • 你修改后没提交，导致后面的场景不成立

8.7 第四步：开始合并，制造冲突

现在在主分支上执行：

git merge conflict-branch

8.7.1 你现在应该看到什么

这时 Git 很可能不会直接合并成功，而是给出类似下面的提示：

CONFLICT (content): Merge conflict in conflict-demo.txt
Automatic merge failed; fix conflicts and then commit the result.

不同 Git 版本的文案可能略有差异，但关键信息通常有两点：

  • 某个文件发生了冲突
  • 自动合并失败，需要你手动处理

8.7.2 这一步怎样算成功

算成功：

  • 你真的制造出了一个冲突
  • 你知道这不是仓库损坏，而是 Git 在等你决定内容

算失败：

  • 你本来想做冲突实验，却没有出现冲突
  • 你一看到报错就以为整个仓库坏掉了

如果没有冲突，多半是因为你们改的不是同一位置，或者某一边的修改没提交成功。

8.8 第五步：看懂冲突标记

现在打开 conflict-demo.txt，你通常会看到类似下面的内容：

第一行：Git 可以帮助我们管理修改。
<<<<<<< HEAD
第二行：这是主分支版本的改动。
=======
第二行：这是分支版本的改动。
>>>>>>> conflict-branch

这几个标记一定要看懂：

  • <<<<<<< HEAD 下面的是你当前分支的版本
  • ======= 是分隔线
  • >>>>>>> conflict-branch 下面的是被合并进来的那个分支版本

也就是说，这不是乱码，而是 Git 把两个候选版本都摆出来让你选。

8.8.1 这一步怎样算成功

算成功：

  • 你能指出哪一段来自当前分支
  • 你能指出哪一段来自另一条分支
  • 你知道真正需要留下的是“处理完后的最终文本”，不是这些标记本身

算失败：

  • 你把这些标记也一起保留进最终文件
  • 你没搞清楚哪一边是谁的修改

8.9 第六步：手动解决冲突

现在你要做的，不是选边站这么简单，而是决定最终文件到底该长什么样。

比如你决定把两边意思合起来，改成：

第一行：Git 可以帮助我们管理修改。
第二行：这里同时吸收主分支和分支版本的改动。

注意：解决冲突时，你要把那几行标记全部删掉，只留下你真正想保留的结果。

改完以后执行：

git status
git add conflict-demo.txt
git status

8.9.1 你现在应该看到什么

冲突刚出现时，git status 往往会显示类似：

  • both modified
  • unmerged paths

当你把文件内容整理好，并且重新 git add 以后，它会从“未合并状态”进入“准备提交”的状态。

8.9.2 这一步怎样算成功

算成功：

  • 文件里已经没有冲突标记
  • 最终文本是你确认过的版本
  • git add 之后，Git 知道这个冲突已经被你处理了

算失败：

  • 你只是删掉了一边内容，但没认真确认最后文本
  • 文件里还残留 <<<<<<<、=======、>>>>>>>
  • 你以为保存文件就算完成，实际上还没 git add

最后一点非常常见。解决冲突不是只改文件，还要重新告诉 Git：“我已经处理完了。”

8.10 第七步：完成这次合并

处理完以后，再执行：

git commit
git log --oneline -5

很多时候，Git 会自动生成一条默认的合并说明。你可以保留它，也可以顺手改得更清楚一点。

8.10.1 你现在应该看到什么

如果流程顺利：

  • 合并提交已经建立起来
  • git log --oneline -5 里能看到这次 merge 的结果
  • 工作区会回到干净状态

8.10.2 这一步怎样算成功

算成功：

  • 你真的完成了整个冲突处理流程
  • 合并后的仓库状态正常
  • 历史里能看见这次合并

算失败：

  • 你以为改完文件就结束了，没有做最后的提交
  • 你解决完冲突后，没有检查最终历史

8.11 为什么解决冲突后还要再提交

因为“解决冲突”本身就是一次新的决定。

在这个决定里，你做的事不是简单选一边，而是明确告诉 Git：

  • 这两条线上冲突的地方，我已经整理好了
  • 现在请把这个结果记进历史

所以最后那笔提交，不是多余步骤，而是整个合并流程的收尾。

8.12 怎样减少无谓冲突

冲突不可能完全消失，但很多无谓冲突是可以减少的。

8.12.1 尽量不要长期让分支悬着不合

分支拖得越久，主线变化越多，后面碰撞的机会就越大。

8.12.2 一次只处理一个清楚主题

如果一笔分支里同时混着重构、修错字、加新内容，后面就更容易和别人的修改打架。

8.12.3 经常同步主线变化

如果你知道主线已经变了很多，不要拖到最后一天才去合并。

8.12.4 改动前先看最近别人动了哪里

尤其是多人协作时，先看日志和最近提交，能减少很多无谓撞车。

8.13 遇到冲突时最容易犯的错

8.13.1 看到冲突标记就慌

其实它只是提示，不是损坏。

8.13.2 没看清内容就随手删掉一边

有时两边修改都值得保留，不能机械地只留一边。

8.13.3 改完文件却忘了 git add

这样 Git 还不知道你已经处理完。

8.13.4 解决完不检查最终文件

冲突处理的目标不是“让 Git 不报错”，而是让最终内容真的正确。

8.14 本章练习

8.14.1 练习 1：亲手制造一次冲突

按本章步骤，在测试仓库里完整做一遍：

  1. 准备基础文件
  2. 切分支并修改同一行
  3. 回主分支修改同一行
  4. 执行 git merge

算成功：

  • 你真的制造出了冲突
  • 你没有把冲突误解成仓库损坏

算失败：

  • 你没有在同一位置制造不同修改，结果练习没形成冲突

8.14.2 练习 2：手动解决并完成提交

  1. 打开冲突文件
  2. 删除标记，写出最终版本
  3. git add <文件>
  4. git commit
  5. git log --oneline -5

算成功：

  • 冲突标记全部消失
  • 最终内容是你确认过的
  • 历史里出现合并结果

算失败：

  • 文件里还残留冲突标记
  • 你改完了却忘了 git add 或 git commit
