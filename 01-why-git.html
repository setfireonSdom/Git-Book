1  为什么需要 Git

很多人第一次接触 Git，不是因为突然对版本控制有兴趣，而是因为文件开始变多了，项目开始改来改去，光靠文件夹备份已经不够用了。

这一章要解决的问题很简单：Git 到底在帮我们省掉什么麻烦？如果不用 Git，平常又会怎么出错？把这件事想明白，后面那些命令才不会显得像一套硬背的咒语。

本章会回答几个问题：

  • 为什么「final」「final2」「final-真的最终版」不是好办法
  • Git 和普通云盘、文件夹备份有什么不同
  • 为什么写代码的人要学，写文档的人其实也该学
  • Git 最适合解决哪些问题，不适合解决哪些问题

读完这一章，你应该能理解：Git 不是为了增加步骤，而是为了让修改变得可追踪、可回退、可协作。

1.1 你可能已经这样做过

如果你以前没有学过 Git，管理文件时大概用过这些办法：

  • 直接覆盖旧文件
  • 另存一个新文件，名字叫 report-final.docx
  • 改完以后又存一个 report-final-2.docx
  • 担心出错，再复制一个 report-final-真的最终版.docx
  • 想起上周删过一段文字，却不知道去哪里找回来

这些办法不是完全没用。项目很小、修改很少、也只有自己一个人动手时，勉强还能撑住。问题在于，只要修改次数一多，下面几件事就会一起冒出来：

  • 你不知道哪个文件才是最新的
  • 你知道文件变了，却说不清到底哪里变了
  • 你想回到昨天的状态，却没有可靠的办法
  • 两个人同时改同一个文件时，后合并的人很容易把前面的修改盖掉

Git 要解决的，就是这些问题。

1.2 Git 到底在管什么

先把一句话放在这里：Git 管的不是“文件夹里有没有文件”，而是“这个项目是怎么一步步改成现在这个样子的”。

这句话很重要，因为很多人一开始会把 Git 误会成：

  • 一个高级云盘
  • 一个自动备份工具
  • 一个专门给程序员用的东西

它和这些东西都沾边，但都不完全一样。

云盘擅长同步文件，让你在不同设备上都能拿到同一份内容。Git 擅长记录修改，让你知道谁在什么时候改了什么，还能回到之前的状态。两者可以一起用，但负责的事情不同。

普通备份更像“按时间留副本”。Git 更像“按修改留历史”。你不是被迫保存一整堆 final 文件，而是把每一次关键变动组织成一条清楚的历史线。

这里有个很关键的转变。没用 Git 时，我们常常只是“把当前结果存下来”；至于中间怎么改过、为什么这样改，时间一久就说不清了。用了 Git 以后，你开始管理的就不只是结果，还包括通往这个结果的过程。

1.3 为什么文件夹备份很快就不够用

先看一个很常见的场景。

你在做一个个人网站，第一天存成 index.html。第二天你准备大改首页文案，怕翻车，于是复制一份，叫 index-new.html。改了一半觉得不太好，又回去修旧稿。第三天你决定融合两个版本，结果两个文件都改过了，你已经不记得：

  • 哪一份才是最新的
  • 哪一份删过一段更好的话
  • 哪一份加入了新的例子

这时真正麻烦的，不是文件太多，而是“变化没有被结构化地记录下来”。

Git 的第一个价值，就是让你不再用文件名承担“版本管理”这件事。

很多人第一次真正感受到 Git 的价值，不是在建仓库那一刻，而是在几天后回头看时，发现自己终于不用靠猜来回忆“那版到底改了什么”。

1.4 Git 和云盘、备份的区别

这一点最好一开始就分清楚。

工具 最擅长的事 不擅长的事
云盘 同步文件、跨设备访问、共享文件 清楚比较修改历史、组织开发流程
手动备份 留下某个时间点的副本 难以持续维护，版本很快混乱
Git 记录修改、比较差异、回退历史、支撑协作 不能替代日常网盘，也不是所有二进制文件都适合用它管

所以比较稳的理解不是“用了 Git 就不用备份”，而是：

  • Git 负责管修改历史
  • 云盘负责管文件同步
  • 备份负责防止整体丢失

这三个角色分清楚以后，后面很多误会会少很多。比如有人会问：“我都放网盘了，为什么还要 Git？”这个问题其实就像在问：“我已经把纸张装进文件盒了，为什么还要给文稿写修订记录？”两者不是互相替代，而是在处理不同问题。

1.5 为什么不只写代码的人该学 Git

很多人一听 Git，就先想到程序员。其实只要你做的是“会反复修改，而且希望看得清历史”的工作，Git 都有价值。

例如：

  • 写论文、书稿、讲义
  • 维护网站文案
  • 写配置文件
  • 记录数据处理脚本
  • 和别人一起改同一份文档

像个人网站、小型文档站点、课程笔记仓库这类项目，都很适合用 Git。它们的文件大多是文本文件，修改会不断累积。如果不用 Git，时间一久，最容易出现的不是“文件丢了”，而是“文件还在，但已经说不清怎么变成现在这样”。

这一点对持续打磨的内容尤其明显。页面不是一次写完的，而是一轮一轮往前推：改标题、换文案、补说明、挪顺序、重写某一段。要是这些变化都没被好好记住，写到后面就会很难判断“这版为什么比上一版更好”，也很难放心回退。

1.6 Git 最有用的四件事

如果要把 Git 的价值压缩成最核心的四点，可以这样记。

1.6.1 一是看得见变化

你可以知道这次到底改了什么，而不是只知道“文件好像不一样了”。

1.6.2 二是回得去之前

你改坏了，或者实验一条路以后想退回来，不需要靠运气找旧文件。

1.6.3 三是能把一次修改说清楚

你不是把一整个杂乱状态扔进去，而是能整理成一笔一笔提交。以后回头看历史时，才会知道每一笔改动是在解决什么问题。

1.6.4 四是能让多人协作不那么乱

没有 Git 时，两个人同时改一份东西，通常靠聊天沟通“你先别动，我改完发你”。有了 Git 以后，协作会变得更可控，因为每个人的修改都有明确记录，也有合并机制。

1.7 Git 不是什么

学 Git 时，早点知道它的边界也很重要。

Git 不是：

  • 自动帮你写好提交说明的老师
  • 永远不会出错的保险箱
  • 所有文件都适合直接放进去的仓库
  • 学会几条命令以后就不用理解流程的魔法工具

再补一句更实际的话：Git 也不是让你从此再也不会乱。它只是把“乱”变得可见、可整理、可回头。这个差别看起来不大，用起来却非常大。

尤其是最后一点。Git 的难，不全在命令，而在于你要知道自己现在是在处理哪一层：工作区、暂存区、历史，还是远端。这个问题下一章会正式展开。

1.8 先做一个很小的体验

这一章先不急着上来就学完整命令，但可以先做一个很小的观察实验。目的不是立刻学会 Git，而是亲眼看到“有记录”和“没记录”的差别。

1.8.1 小实验：用普通文件夹管理修改

你可以在桌面随便建一个目录，比如 git-demo-without-git，然后做下面的事：

  1. 建一个 notes.txt
  2. 写入第一版内容
  3. 过几分钟改掉其中两三句
  4. 再删掉一段
  5. 试着回答下面几个问题
  • 第二次到底改了哪几句
  • 删掉的那一段原来写了什么
  • 如果你只想保留其中一处修改，怎么做

只靠普通文件夹，你通常做不到很轻松地回答这些问题。

1.8.2 如果现在就用 Git，成功会长什么样

下一章开始我们会真正动手。你先记住一个判断标准：Git 用起来顺不顺，不是看命令背得多不多，而是看你能不能明确回答下面几件事：

  • 我现在改了哪些文件
  • 哪些修改准备提交，哪些还没准备提交
  • 上一次提交记录了什么
  • 如果改坏了，我能退到哪里

只要你能稳定回答这四个问题，Git 就已经开始真的帮上忙了。

这也是这本书后面每一章反复在做的事。我们不是为了多学几条命令，而是在不断练这四个判断：现在改了什么，准备交什么，历史里已经有什么，出错时能退到哪里。

1.9 这一章的结论

这一章还没进入细命令，但有三件事应该先钉牢。

第一，Git 不是为了把流程变复杂，而是为了让修改历史清楚下来。

第二，Git 最重要的价值，不是“多存一份”，而是让你看得见变化、回得去之前、说得清每次修改在做什么。

第三，如果你以后在 Git 里迷路，通常不是因为你不聪明，而是因为你还没把它的心智模型看清楚。下一章我们就专门解决这个问题。

如果这一章读完，你已经从“Git 很厉害，但我不知道它到底在管什么”变成“Git 管的是修改过程，而不只是文件结果”，那这一章的任务就完成了一大半。

1.10 本章练习

这一章的练习不要求你已经会 Git，只要求你开始用“版本管理”的眼光看手头项目。

1.10.1 练习 1：找一个你正在维护的文件

可以是下面这些类型之一：

  • 一篇文档
  • 一个 Markdown 文档
  • 一个脚本
  • 一份配置文件

然后回答：

  1. 它最近一周改过几次
  2. 你能不能清楚说出每次改了什么
  3. 如果要回到前天的版本，你现在有没有把握做到

如果这三个问题里有两个你答不稳，就说明这个文件已经很适合交给 Git 管。

1.10.2 练习 2：判断成功和失败

先不用操作 Git，只判断自己的现状。

算成功：

  • 你已经能用自己的话说清楚 Git 在解决什么问题
  • 你知道 Git 不等于 GitHub
  • 你知道手动备份和版本管理不是同一回事

算失败：

  • 你仍然觉得 Git 只是“程序员用来上传代码的网站工具”
  • 你觉得版本管理等于“多存几个副本”
  • 你还说不清为什么需要“修改历史”而不只是“最新文件”
