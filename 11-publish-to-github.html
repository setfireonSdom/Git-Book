11  把本地项目放到 GitHub

这一章会带你走完第一次把项目发布到 GitHub 的流程。从本地已有仓库开始，到建立远端仓库，再到完成第一次推送，整个过程尽量按真实顺序来。

你会在这一章学到：

  • 怎么在 GitHub 建立一个新仓库
  • 怎么把本地仓库连到远端
  • 第一次 push 时会发生什么
  • 以后怎么同步本地和远端

这一章结束后，你手上的项目就不只存在于自己电脑里，而是已经能放到线上继续协作。

11.1 这一章真正要做成什么

这一章的目标很具体。做完之后，你手上的本地仓库应该变成这样：

  • 本地已经有正常提交历史
  • GitHub 上已经有一个对应远端仓库
  • 你的本地仓库已经连上那个远端
  • 你已经成功完成第一次 push

如果这四件事都成立，就说明你不是“知道怎么上传”，而是真的把本地仓库接到了 GitHub。

11.2 开始前先确认本地条件

先在本地仓库里执行：

git status
git log --oneline -5
git remote -v

11.2.1 你现在应该看到什么

开始之前，比较理想的状态是：

  • git status 显示工作区干净，或者至少你自己知道还有哪些改动
  • git log --oneline -5 能看到已有历史
  • git remote -v 还没有任何输出，说明这个仓库可能还没连上远端

11.2.2 这一步怎样算成功

算成功：

  • 你确定自己手上有一个正常工作的本地仓库
  • 你知道这个仓库目前有没有远端

算失败：

  • 你连本地提交历史都还没有，就急着接 GitHub
  • 你其实已经配过远端，却没先看清楚

如果 git remote -v 已经有输出，不代表不能继续，但你得先看清楚你是不是已经连过某个远端，避免重复配置。

11.3 第一步：在 GitHub 上建立一个新仓库

这一步主要在网页上完成。进入 GitHub 后，新建一个仓库。

第一次接本地仓库时，比较稳的做法是：

  • 先建一个空仓库
  • 不要勾选自动生成 README
  • 不要先加 .gitignore
  • 不要先加 license

这样做的原因很简单：你本地已经有一套历史了。如果远端一开始就多出自己的初始提交，第一次对接时反而更容易让初学者混乱。

11.3.1 这一步怎样算成功

算成功：

  • 你在 GitHub 上得到了一个新的空仓库页面
  • 页面上能看到仓库地址

算失败：

  • 你先让 GitHub 自动生成了额外文件，却没意识到远端已经有自己的初始提交
  • 你还没确认仓库地址，就匆忙回到终端继续

11.4 第二步：把远端地址连到本地仓库

GitHub 新仓库页面通常会给你一个地址，常见形式像这样：

https://github.com/yourname/your-repo.git

现在在本地仓库里执行：

git remote add origin https://github.com/yourname/your-repo.git
git remote -v

11.4.1 git remote add origin 在做什么

它的意思是：

  • 给这个本地仓库登记一个远端
  • 这个远端的名字叫 origin
  • 对应的地址是你刚才 GitHub 仓库的地址

origin 只是常见名字，不是魔法关键字，但大多数教程和项目都会这么叫。

11.4.2 你现在应该看到什么

执行完 git remote -v，通常会看到类似：

origin  https://github.com/yourname/your-repo.git (fetch)
origin  https://github.com/yourname/your-repo.git (push)

11.4.3 这一步怎样算成功

算成功：

  • 本地仓库已经记住了一个名为 origin 的远端
  • git remote -v 能正确显示地址

算失败：

  • 你把远端地址粘错了
  • 你重复添加同名远端，结果 Git 报错
  • 你没有检查远端地址是否真的是目标仓库

如果提示 remote origin already exists，不要慌，先说明这个仓库里已经有个叫 origin 的远端了。先用 git remote -v 看清楚它指向哪里，再决定下一步。

11.5 第三步：确认当前分支名

在第一次推送前，最好先确认自己当前分支叫什么。

执行：

git branch

11.5.1 你现在应该看到什么

你通常会看到：

* main

或者：

* master

11.5.2 这一步怎样算成功

算成功：

  • 你知道当前分支到底叫 main 还是 master
  • 你不会在推送时凭印象乱写分支名

算失败：

  • 你默认以为一定是 main
  • 结果命令里写错分支名，后面推送报错

11.6 第四步：第一次把本地历史推到 GitHub

现在可以开始第一次推送了。假设当前分支是 main，执行：

git push -u origin main

如果你的当前分支是 master，就把最后一个 main 改成 master。

11.6.1 -u 在做什么

这里的 -u 很值得顺手记住。它的作用是：

  • 把本地当前分支和远端对应分支建立跟踪关系

这样以后你再在这个分支上 git push 或 git pull，通常就不用每次都把远端名和分支名写全。

11.6.2 你现在应该看到什么

第一次推送成功后，终端通常会告诉你：

  • 发送了哪些对象
  • 新分支已经推到远端
  • 当前分支开始跟踪远端对应分支

然后你回到 GitHub 页面刷新，通常会看到：

  • 仓库不再是空的
  • 你的文件和提交历史已经出现在网页上

11.6.3 这一步怎样算成功

算成功：

  • git push -u origin main 或 master 成功执行
  • GitHub 页面上已经能看到你的项目文件
  • 远端分支和本地分支已经建立跟踪关系

算失败：

  • 推送报错，提示分支名不对
  • 推送报错，提示认证失败
  • 命令执行完了，但 GitHub 页面刷新后还是空的

最后一种情况说明：你以为推上去了，但实际上并没有成功完成远端更新。

11.7 第五步：验证这次连接是不是真的建好了

第一次推送成功以后，最好再做几个检查，不要只看一条命令是否报错。

执行：

git remote -v
git branch -vv
git status

11.7.1 你现在应该看到什么

  • git remote -v：还能看到远端地址
  • git branch -vv：当前分支后面通常会显示它正在跟踪哪个远端分支
  • git status：通常会告诉你当前分支和远端分支已经同步

11.7.2 这一步怎样算成功

算成功：

  • 你确认本地和远端的连接已经稳定建立
  • 你知道当前分支在跟踪哪一个远端分支

算失败：

  • 你只知道“刚才好像推成功了”
  • 但说不清当前分支和远端是什么关系

11.8 如果第一次推送时遇到常见问题

11.8.1 情况 1：分支名写错

比如你当前分支其实叫 master，却执行了：

git push -u origin main

这时 Git 往往会报找不到对应本地分支。

更稳的做法始终是先跑一次：

git branch

看清楚再推。

11.8.2 情况 2：远端已经不是空仓库

如果你在 GitHub 建仓库时已经让它自动生成了 README 或别的初始文件，远端就不再是空的。

这时第一次推送可能不会像空仓库那样简单。对初学者来说，最稳的方式还是一开始就先建空仓库，减少第一轮混乱。

11.8.3 情况 3：认证失败

如果 GitHub 拒绝了你的推送，常见原因包括：

  • 账号没有登录好
  • 权限不对
  • 认证方式没设置好

这类问题不是 Git 逻辑本身出错，而是本地和 GitHub 账号之间的认证还没配好。

11.9 第六步：以后日常怎么同步

第一次连接完成以后，后面的日常节奏就会简单很多。最常见的顺序是：

  1. 本地继续修改
  2. git add
  3. git commit
  4. git push

如果别人也在远端改了东西，或者你换了一台机器继续工作，常见流程就会变成：

  1. 先 git pull 或先 git fetch
  2. 看远端有没有新变化
  3. 再决定怎么整合

也就是说，GitHub 的加入并没有推翻前面学过的 Git 节奏，只是在本地历史之外，多了一层和远端同步的动作。

11.10 本章最值得确认的不是“会不会推”，而是“推的是什么”

这一章里最容易出现的误会是：

  • 以为 push 推的是当前文件夹现状

实际上，push 推的是已经提交好的历史。所以如果本地工作区还在乱、提交边界还没整理好，就算推上去了，远端也不会变得更清楚。

这也是为什么前面几章要先把本地节奏练稳。GitHub 不是帮你替代整理过程，而是把你已经整理好的历史同步出去。

11.11 本章练习

11.11.1 练习 1：检查一个本地仓库有没有远端

执行：

git remote -v

算成功：

  • 你能判断这个仓库现在有没有远端
  • 你能看懂 origin 和地址分别是什么

算失败：

  • 你看到输出，却不知道它表示什么

11.11.2 练习 2：把本地仓库连到远端

  1. 在 GitHub 建一个空仓库
  2. git remote add origin <远端地址>
  3. git remote -v

算成功：

  • 远端已经正确登记在本地仓库里

算失败：

  • 地址写错，或者重复添加远端却没发现

11.11.3 练习 3：完成第一次推送

  1. git branch
  2. git push -u origin <当前分支名>
  3. 刷新 GitHub 页面
  4. git branch -vv

算成功：

  • GitHub 页面已经能看到你的项目
  • 当前分支已经在跟踪远端分支

算失败：

  • 终端命令跑了，但远端其实没有成功更新
  • 你推错分支，或者根本没确认当前分支名
